Symmetric Cryptography Рђћ How It Works

One shared secret key encrypts and decrypts. The math transforms plaintext through multiple rounds so thoroughly that reversing it without the key is computationally impossible.

AES Рђћ How Each Round Works

FIPS 197 ┬и Block Cipher

AES operates on a 4├Ќ4 grid of bytes called the state. Every round applies four transformations in sequence. AES-128 does 10 rounds; AES-256 does 14 rounds.

SubBytes Рђћ Confusion

Every byte in the 4├Ќ4 state is replaced by a fixed lookup table called the S-Box. This substitution is non-linear Рђћ it makes the relationship between key and ciphertext impossible to express as a simple equation.

# S-Box substitution (simplified) state[r][c] = SBOX[ state[r][c] ] # each byte Рєњ new byte # e.g. 0x53 Рєњ 0xED (from fixed S-Box table) # Non-linear Рђћ breaks algebraic attacks
ShiftRows Рђћ Diffusion (horizontal)

Each row of the 4├Ќ4 state is shifted left by 0,1,2,3 positions respectively. Bytes from one column spread into different columns Рђћ so one changed byte affects the whole block.

# Row 0: no shift [a0 a1 a2 a3] # Row 1: shift left 1 [b1 b2 b3 b0] # Row 2: shift left 2 [c2 c3 c0 c1] # Row 3: shift left 3 [d3 d0 d1 d2]
MixColumns Рђћ Diffusion (vertical)

Each column of 4 bytes is multiplied by a fixed matrix in GF(2РЂИ) Рђћ the Galois Field. One changed byte now affects all 4 bytes in the column. Combined with ShiftRows, changing 1 input byte affects all 16 output bytes after 2 rounds (avalanche effect).

# Matrix multiplication in GF(2^8) [s'0] [2 3 1 1] [s0] [s'1] = [1 2 3 1] [s1] (mod irreducible polynomial) [s'2] [1 1 2 3] [s2] [s'3] [3 1 1 2] [s3]
AddRoundKey Рђћ Key Injection

The 128-bit round key (derived from the master key via Key Schedule) is XORed with the entire state. XOR is reversible only if you know the key. This is the only step that uses the secret key directly.

# XOR each byte of state with round key state[r][c] = state[r][c] XOR roundKey[r][c] # Repeat all 4 steps for 10/12/14 rounds # Final round: SubBytes + ShiftRows + AddRoundKey (no MixColumns)

ECC Рђћ Elliptic Curve Mechanism

Points on an elliptic curve y┬▓ = x┬│ + ax + b form a group. "Adding" two points follows a geometric rule Рђћ draw a line through them, find where it hits the curve, reflect.

New math problems that neither classical nor quantum computers can solve efficiently. LWE (Learning With Errors) is the core foundation of ML-KEM and ML-DSA.

Step 3 Рђћ Why Quantum Cannot Solve LWE

  • Grover's algorithm can search in РѕџN time Рђћ but for LWE with n=768, the search space is 2^768. Grover reduces this to 2^384 Рђћ still astronomically infeasible.
  • BKZ lattice reduction is the best known classical lattice attack Рђћ but it runs in exponential time in the security parameter. ML-KEM-768 is designed so BKZ would take longer than the age of the universe.

What Is a Lattice?

A lattice is an infinite regular grid of points in n-dimensional space. It is defined by a basis Рђћ a set of vectors from which every lattice point can be reached by integer combinations.

# 2D lattice defined by two basis vectors bРѓЂ = [3, 0] # first basis vector bРѓѓ = [1, 2] # second basis vector # Every lattice point = integer combination P = a┬иbРѓЂ + b┬иbРѓѓ where a,b Рѕѕ Рёц # Examples: [0,0] = 0┬иbРѓЂ + 0┬иbРѓѓ (origin) [3,0] = 1┬иbРѓЂ + 0┬иbРѓѓ [4,2] = 1┬иbРѓЂ + 1┬иbРѓѓ [7,2] = 2┬иbРѓЂ + 1┬иbРѓѓ [2,4] = 0┬иbРѓЂ + 2┬иbРѓѓ # In crypto: n = 256, 512, 768 dimensions # Same idea Рђћ incomprehensibly larger space

Key insight: same lattice, different bases

The same lattice can be described by many different bases. A "good" basis has short, nearly-orthogonal vectors. A "bad" basis has long, nearly-parallel vectors.

Good basis Рєњ easy to navigate the lattice, solve problems efficiently. This is what the private key holder has.

Bad basis Рєњ hard to navigate, hard to solve problems. This is what an attacker sees (the public key).

Lattice cryptography security = the gap between a good basis and a bad basis.

Qubits Рђћ What They Actually Are

A classical bit is 0 or 1. A qubit can be in a superposition Рђћ a combination of 0 and 1 simultaneously Рђћ until it is measured. Measurement collapses it to exactly 0 or 1.

Why superposition is NOT "try all at once"

Quantum algorithms work by constructing interference patterns Рђћ correct answers interfere constructively (amplitudes add), wrong answers interfere destructively (amplitudes cancel). Only Shor's, Grover's, and a few others achieve this elegantly.

Quantum Interference Рђћ The Core Mechanism

Quantum speedup comes from interference Рђћ the same phenomenon that creates light/dark bands when light passes through two slits. Correct answers get amplified; wrong answers cancel out.

Analogy: Imagine dropping a pebble in a pond Рђћ ripples spread in all directions. Now drop two pebbles simultaneously at different spots. Where ripples meet in phase Рђћ they amplify (constructive interference). Where they meet out of phase Рђћ they cancel (destructive interference). Quantum algorithms engineer this so the answer's amplitude grows and all wrong answers' amplitudes shrink to zero.
# Quantum algorithm structure 1. Initialise qubits in superposition of ALL possible inputs 2. Apply quantum gates that encode the problem 3. Use interference to amplify correct answer's amplitude 4. Measure Рєњ get correct answer with high probability # The art is in step 3 Рђћ designing the interference pattern # Only works for problems with exploitable mathematical structure # LWE has no such structure Рєњ quantum interference cannot help

Do not attack the mathematics. Attack the physical implementation Рђћ measuring what the device leaks while it computes. Even a mathematically perfect algorithm can be completely broken by a poor implementation.

Power Analysis Рђћ SPA and DPA

Measure electrical power a device consumes during cryptographic operations. Power consumption correlates with data being processed Рђћ different key bits cause measurably different power patterns.

# Simple Power Analysis (SPA) Рђћ RSA example if bit == 1: square_and_multiply() # more power if bit == 0: square_only() # less power # Single power trace Рєњ full binary key sequence # Differential Power Analysis (DPA) Рђћ statistical # Many traces + correlation with guessed key bits # Works even with noise Рђћ statistics win over thousands of traces # PQC vulnerability: ML-KEM Gaussian sampling # Power proportional to sampled secret polynomial values # FALCON floating-point sampler leaks through power traces
Timing Analysis Рђћ Branch and Cache Timing

If an operation takes different time depending on secret values, an attacker measuring execution time can extract the secret. Differences as small as nanoseconds are exploitable with enough measurements.

Electromagnetic Emission Analysis (EMA)

Every electronic circuit radiates EM signals as it operates. These emissions carry information about data being processed Рђћ measurable from a distance without physical contact.

# EMA attack setup EM probe near device (or antenna metres away) РєЊ Capture EM signal during PQC operation РєЊ Frequency analysis + correlation РєЊ Extract secret polynomial coefficients # Real attack (Ruhr University 2021): # Extracted ML-KEM secret keys from microcontroller # Using EM probes Рђћ no physical contact # Works even on constant-time implementations # EM leakage harder to eliminate than timing
SCA Countermeasures for PQC
  • Masking Рђћ split secret s into random shares: s = s1 + s2. Process each separately. Leaking one share reveals nothing about s.
  • Constant-time code Рђћ all branches and memory accesses take identical time regardless of secret values.
  • Shuffling Рђћ randomise operation order so traces don't align across measurements.
  • Hardware shielding Рђћ Faraday cages, EM shielding in PCB design.
  • Noise injection Рђћ random dummy operations mask real power patterns.
  • Formal verification Рђћ mathematically prove constant-time property of code.
Рџа№ИЈ Masking PQC costs 2-5x more computation and memory than unmasked Рђћ a major challenge for IoT and smartcard deployments already tight on resources.

Fault Injection Attacks (FIA)

Code Layer ┬и Physical

Intentionally cause computational errors during cryptographic operations. The faulty output compared with the correct output reveals secret key information. A single successful fault can break an otherwise secure implementation.

# Fault injection methods Voltage glitch Рєњ CPU skips instruction Clock glitch Рєњ double-execute one step Laser pulse Рєњ flip specific memory bit EM pulse Рєњ corrupt register value Temperature Рєњ cause unpredictable errors # DFA on ML-DSA signing 1. Get correct signature S = sign(M, key) 2. Inject fault Рђћ skip rejection check 3. Get faulty S' = sign(M, key) with error 4. S - S' = f(key, fault) 5. Repeated faults Рєњ full key recovery

FIA on PQC specifically

ML-DSA signing has a rejection step Рђћ if output norm is too large, it retries. If a fault forces it to skip this check and output an invalid signature Рђћ that signature leaks secret key coefficients.

FALCON is particularly vulnerable Рђћ its rejection sampling step, if faulted, can leak the secret NTRU basis directly in a single attack.

Countermeasures: double computation, redundant checks, voltage/clock monitors, physical tamper protection.

The FO Transform Protection

ML-KEM applies the Fujisaki-Okamoto transform Рђћ making the scheme CCA2-secure. During decapsulation, ML-KEM re-encrypts and rejects any ciphertext that doesn't match exactly Рђћ eliminating the decryption oracle.

Critical: If the FO check is bypassed or faulted, the full oracle is exposed. A single missing line of code in the rejection check can compromise the entire key. This is a prime target for fault injection attacks.

BKZ Lattice Reduction Рђћ The Best Known Attack on LWE

Math Layer ┬и Cryptanalysis

Block Korkine-Zolotarev (BKZ) is the most powerful known algorithm for finding short vectors in lattices Рђћ and therefore the best tool for attacking ML-KEM and ML-DSA. Understanding BKZ is understanding the exact hardness assumption PQC relies on.

What BKZ Does Рђћ Basis Reduction

Given a bad lattice basis (long, non-orthogonal vectors Рђћ what an attacker has from the public key), BKZ tries to find a better basis with shorter, more orthogonal vectors. A sufficiently good basis can solve SVP Рђћ finding the shortest vector Рђћ which breaks the LWE problem.

# BKZ high-level algorithm Input: bad basis B (from public key Рђћ long vectors) Output: better basis B' (shorter vectors) For each block of ╬▓ consecutive basis vectors: 1. Project out other dimensions 2. Solve SVP in ╬▓-dimensional subspace 3. Use short vector found to reduce other vectors 4. Repeat until no more improvement # Larger ╬▓ = better reduction = shorter vectors found # But: SVP in ╬▓ dimensions costs 2^(0.292╬▓) # So BKZ-╬▓ costs O(poly(n) ├Ќ 2^(0.292╬▓)) total # Quality vs cost: bigger block = better but slower

The key insight: BKZ doesn't solve LWE directly. It finds a shorter basis, then uses that basis to make LWE easier to solve. If BKZ can find a short enough vector, it can recover the secret s from the noisy equation As + e = b.

Algebraic Attacks Рђћ Exploiting Structure in Polynomial Rings

Math Layer ┬и Structural

ML-KEM and ML-DSA don't use plain LWE Рђћ they use Module LWE (MLWE) over polynomial rings for efficiency. This extra structure is a potential attack surface Рђћ structured problems can sometimes be easier than unstructured ones.

# Plain LWE (no structure) A Рѕѕ Z_q^(m├Ќn), s Рѕѕ Z_q^n b = As + e Рєљ random matrix A # Module-LWE (structured ring) A Рѕѕ R_q^(k├Ќk), s Рѕѕ R_q^k b = As + e Рєљ A over ring R_q = Z_q[x]/(x^n+1) # Ring structure = polynomials multiply efficiently # But ring structure = potential algebraic attack surface # Subfield attacks: exploit ring automorphisms # Twisted embedding attacks: use complex geometry

Why Ring-LWE was chosen despite risk

Plain LWE is more conservative but requires much larger keys (megabytes). Ring-LWE enables the compact 1.2KB keys of ML-KEM. After 15+ years of analysis, no algebraic attack has succeeded against properly parameterised Ring-LWE or Module-LWE.

The choice is a calculated tradeoff: accept slightly more structural risk in exchange for practical key sizes. ML-KEM-768's parameters were specifically chosen to resist all known algebraic attacks with significant margin.

The SIKE Collapse Рђћ A Complete Case Study

Math Layer ┬и Historical Break

SIKE (Supersingular Isogeny Key Encapsulation) was a NIST PQC finalist broken in 62 minutes on a laptop in 2022. It is the clearest proof that math creates AND breaks security Рђћ and the most important warning for all PQC deployments.

What SIKE Was Рђћ Isogeny-Based Cryptography

SIKE based its security on the Supersingular Isogeny Diffie-Hellman (SIDH) problem: given two supersingular elliptic curves E and E/РЪеPРЪЕ connected by a secret isogeny (a special curve homomorphism) ¤є, find ¤є.

# SIKE key exchange (simplified) Alice: pick secret isogeny ¤є_A: E Рєњ E_A Bob: pick secret isogeny ¤є_B: E Рєњ E_B Alice sends E_A and images ¤є_A(P_B), ¤є_A(Q_B) Bob sends E_B and images ¤є_B(P_A), ¤є_B(Q_A) # Shared secret: j-invariant of E_AB Alice: compute ¤є_A(E_B) Рєњ E_AB Bob: compute ¤є_B(E_A) Рєњ E_AB (same curve) # The extra points ¤є(P_B), ¤є(Q_B) were needed # for the math to work out correctly # These extra points are what killed SIKE
The Castryck-Decru Attack Рђћ 62 Minutes

Wouter Castryck and Thomas Decru (KU Leuven, July 2022) found that the auxiliary points ¤є(P_B), ¤є(Q_B) given in the public key provided enough information to reconstruct the secret isogeny using a classical mathematical result Рђћ Kani's theorem about products of elliptic curves.

The auxiliary points were not a bug Рђћ they were a necessary feature of how SIDH was designed to work. The security assumption was simply wrong: the auxiliary points revealed far more about the secret than anyone realised.

Why Six Years of NIST Review Missed It

SIKE was submitted in 2017, survived three rounds of NIST review, and was about to be standardised. How did the break get missed?

  • New math family Рђћ isogeny-based crypto was only ~10 years old at submission. The community analysing it was tiny compared to RSA (50 years) or lattice crypto (25 years).
  • Wrong security model Рђћ reviewers focused on finding isogenies directly. Nobody thought to use Kani's theorem Рђћ a result from a completely different area of algebraic geometry.
  • Auxiliary points not questioned deeply enough Рђћ they were seen as a clever engineering trick, not a security liability. The assumption "they're needed, so they're fine" was never rigorously challenged.
  • Cross-field insight required Рђћ the attack combined isogeny theory with results about abelian varieties and Kani's lemma Рђћ a connection nobody in the isogeny crypto community had made.
Lessons for All PQC Deployments
  • Analysis time matters enormously Рђћ LWE has 25+ years of study by thousands of researchers. ML-KEM is built on much more solid ground than SIKE's 10-year-old foundation.
  • Cryptographic agility is mandatory Рђћ systems must be able to swap algorithms quickly when one breaks. SIKE's collapse would have been catastrophic for systems locked into it.
  • Hybrid mode provides insurance Рђћ X25519 + ML-KEM means if ML-KEM falls like SIKE, X25519 still protects the session (classically). Hybrid is not paranoia Рђћ it is engineering prudence.
  • Features can be vulnerabilities Рђћ the auxiliary points were a necessary feature, not an optional extra. Security features must be scrutinised as potential attack surfaces, not just the algorithm core.
  • Math creates AND breaks security Рђћ the attack used pure classical mathematics. No quantum computer, no implementation flaw, no side-channel. The security assumption was simply wrong.

The uncomfortable truth

We cannot be certain ML-KEM will not face a SIKE-style collapse. What we can say: LWE has survived 25 years of analysis by hundreds of researchers across dozens of countries, including adversarial nation-states who have every incentive to break it and would benefit enormously from doing so silently. The security margin is orders of magnitude larger than SIKE's was. This is the strongest evidence available Рђћ not a guarantee.

Timing Attack on ML-KEM Рђћ Concrete Example

Code Layer ┬и Timing SCA

ML-KEM's FO transform requires a conditional check Рђћ if the re-encrypted ciphertext doesn't match, reject. A naive implementation that returns early on mismatch creates a timing oracle that leaks the secret key.

# VULNERABLE ML-KEM decapsulation def decaps_vulnerable(sk, c): m_prime = decrypt(sk, c) c_prime = encrypt(pk, m_prime) if c != c_prime: # Рєљ timing leak here return reject_early() # returns FAST return KDF(m_prime) # returns SLOW # Attacker measures: fast=mismatch, slow=match # Crafts ciphertexts near boundary # Binary search on secret key bits # SAFE ML-KEM decapsulation def decaps_safe(sk, c): m_prime = decrypt(sk, c) c_prime = encrypt(pk, m_prime) match = ct_compare(c, c_prime) # constant time # Always compute both paths result_real = KDF(m_prime) result_dummy = random_bytes() return ct_select(match, result_real, result_dummy)

Why this is hard to get right

Constant-time programming is extremely unintuitive. Compilers optimise code Рђћ and "optimisation" often means "reintroduce the branch the programmer removed." A compiler can silently turn a constant-time compare back into an early-exit compare during optimisation.

The official ML-KEM reference implementation uses carefully verified constant-time operations. But every port to a new platform, language, or compiler must re-verify this property independently.

Real consequence: PQC-TLS timing attacks could leak session keys across the network Рђћ no physical access required.

The FO Transform Рђћ How ML-KEM Blocks the Oracle
# FO Transform in ML-KEM decapsulation def decaps_fo(sk, c): # Step 1: decrypt (get candidate message) m_candidate = decrypt(sk.s, c) # Step 2: re-encrypt using same randomness derivation r_derived = H(m_candidate || sk.H_pk) c_check = encrypt(pk, m_candidate, r_derived) # Step 3: constant-time compare if ct_equal(c, c_check): return KDF(m_candidate || r_derived) else: return KDF(sk.z || c) # random-looking output # Why this blocks the oracle: # Attacker's crafted (u*, v*) will NOT re-encrypt # to itself Рђћ it was not produced honestly # So the FO check ALWAYS fails for crafted ciphertexts # Attacker always gets KDF(z||c) Рђћ random, no info # Oracle completely blind Рђћ no failure signal
Рџа№ИЈ If the FO check is skipped, faulted, or implemented incorrectly Рђћ the full decryption oracle is exposed and the key can be recovered. This single check is the critical security boundary in ML-KEM.

Power Analysis on FALCON Рђћ The Gaussian Sampler Problem

Code Layer ┬и Power SCA

FALCON's signing algorithm requires sampling from a discrete Gaussian distribution over NTRU lattice vectors. This Gaussian sampler is the hardest part of FALCON to implement securely Рђћ and the most powerful SCA target.

# FALCON Gaussian sampler (simplified) To sign message M: 1. Compute target vector t from M 2. Sample z from Gaussian distribution centred at lattice vector near t using secret NTRU basis (f, g, F, G) # The sampler uses floating-point arithmetic # to compute Gaussian probabilities # Different secret basis vectors Рєњ # Different floating-point operations Рєњ # Different power consumption pattern # Power trace Рєњ floating-point op sequence # Рєњ partial information about (f, g) # Рєњ after enough signatures Рєњ full key recovery

Why FALCON is harder to protect than ML-KEM

ML-KEM's operations are polynomial arithmetic Рђћ implementable in constant time using NTT with careful coding. FALCON's Gaussian sampler uses floating-point operations whose timing inherently depends on the values being computed Рђћ making true constant-time extremely difficult.

NIST's FIPS 206 specification includes guidance on implementing FALCON securely, but practical constant-time Gaussian sampling remains an active research problem. For high-security hardware, ML-DSA is often preferred over FALCON precisely for this reason.

Session 3 ┬и Section 03

Breaking the Protocol Layer

Even perfect math and perfect code can be defeated by bad protocol design. These attacks exploit how systems negotiate, combine, and use cryptographic primitives Рђћ not the primitives themselves.

Hybrid PQC Stripping Attack

Protocol Layer ┬и Active Threat

When a TLS client offers a hybrid key share (X25519 + ML-KEM-768), a MitM attacker can strip the ML-KEM component Рђћ leaving only the classical X25519 exchange. The session degrades to a classically-only exchange vulnerable to HNDL.

# Hybrid stripping attack flow Client Рєњ Server: ClientHello key_shares: [X25519_key_share, MLKEM768_key_share] supported_groups: [X25519MLKEM768, X25519, P-256] MitM intercepts and modifies: key_shares: [X25519_key_share] Рєљ ML-KEM stripped supported_groups: [X25519, P-256] Рєљ PQC groups removed Server sees: only classical key shares offered Server responds: ServerHello with X25519 selected # Result: session uses X25519 only # Vulnerable to Shor's algorithm retroactively # Both parties believe negotiation was honest # Neither can detect the stripping without binding # Defence: Transcript binding # TLS 1.3 Finished message is MAC over full handshake # Modified ClientHello Рєњ MAC mismatch Рєњ connection drops # But: does the server enforce minimum PQC requirement? # If server accepts X25519-only Рєњ attack succeeds silently

Defence Рђћ Server-side enforcement

TLS 1.3's Finished MAC protects against transcript modification Рђћ any stripping causes a handshake failure detectable by both parties. However, the attack succeeds if the server is configured to accept classical-only fallback. The defence is server policy: reject any key exchange that does not include a PQC component. This is still not widely enforced Рђћ making hybrid stripping an active deployment risk during the transition period.

During the migration period, servers will offer both classical (RSA/ECDSA) and PQC (ML-DSA) certificates. An attacker can force selection of the classical certificate Рђћ restoring vulnerability to Shor's for certificate authentication.

Why this matters even with ML-KEM

If the key exchange uses ML-KEM but the certificate is ECDSA Рђћ a quantum attacker can forge the server's certificate and impersonate the server. The client establishes a quantum-safe session with what it believes is the real server Рђћ but is actually the attacker.

Defence: Use hybrid certificates (both ECDSA and ML-DSA signatures). Clients supporting PQC must reject classical-only certificates from servers that previously offered PQC Рђћ similar to HSTS pin enforcement.

Breaking the Human and System Layer

The most reliable attacks throughout history have targeted people and operational processes Рђћ not algorithms. PQC changes the math but not human nature.

Weak RNG Рђћ The Silent Key Killer

Human Layer ┬и System Flaw

Every PQC scheme requires high-quality randomness. A weak or predictable RNG makes all other security irrelevant Рђћ keys become guessable regardless of the algorithm's mathematical strength.

RNG requirements for PQC

ML-KEM requires 256 bits of true randomness for key generation and encapsulation. This must come from a CSPRNG seeded by genuine hardware entropy (CPU RDRAND, /dev/urandom after sufficient entropy pool, hardware TRNG). On embedded devices, dedicated hardware RNG chips are mandatory for PQC deployments.

Key Management Failures

Human Layer ┬и Operational

PQC keys are larger than classical keys Рђћ ML-KEM private keys are 2,400 bytes vs 32 bytes for Ed25519. This creates new operational challenges and new failure modes.

  • Key backup failures Рђћ larger keys mean larger backups. Teams accustomed to 32-byte Ed25519 seeds may make mistakes with 2.4KB ML-KEM private keys Рђћ truncating, corrupting, or storing insecurely.
  • HSM compatibility Рђћ many HSMs were built for RSA/ECC key sizes. PQC keys may not fit in existing HSM key slots or may require firmware updates that introduce new vulnerabilities.
  • Key rotation neglect Рђћ PQC migration often treated as "one-time task." But key rotation policies must continue Рђћ a compromised ML-KEM private key is just as dangerous as a compromised RSA key.
  • Certificate lifecycle confusion Рђћ dual-algorithm certificates (ECDSA + ML-DSA) have two separate expiry and renewal cycles. Teams may renew one and forget the other Рђћ leaving a hybrid cert with one expired component.
  • Logging private keys Рђћ debugging PQC implementations sometimes results in private key material being written to logs. ML-KEM's 2.4KB private key is large enough to be "obviously a key" Рђћ but it has happened with smaller keys.
  • Supply chain attacks Рђћ PQC libraries (libpqc, liboqs, BouncyCastle PQC) are relatively new. A compromised PQC library is far more dangerous than a compromised classical library Рђћ fewer eyes have reviewed the code.

The Dual_EC_DRBG Lesson Рђћ Backdoored Standards

Human Layer ┬и Historical Warning

In 2006 NIST standardised Dual_EC_DRBG Рђћ a random number generator that NSA had secretly backdoored. Anyone using it generated "random" keys that NSA could predict. This remained hidden until Snowden documents confirmed it in 2013.

# Dual_EC_DRBG backdoor (simplified) Algorithm uses two curve points: P and Q "Random" output = x-coordinate of (output * Q) If NSA knows: k such that Q = k*P Then given output, NSA can compute: next_state = output * k Рєљ predicts all future values # NSA chose P and Q Рђћ and knew k # All randomness generated was predictable by NSA # Any key, nonce, IV generated was compromised # Juniper VPN used Dual_EC Рєњ NSA backdoor active

Could this happen to PQC?

NIST's PQC process was more transparent than Dual_EC_DRBG Рђћ public submissions, public review, international participation. But the concern is legitimate and has been raised by researchers.

The LWE parameters (q, n, error distribution ¤Ѓ) were chosen by NIST based on NIST's analysis. If any parameters were subtly weakened Рђћ reducing security margins below stated levels Рђћ this would be extremely difficult to detect without independently verifying the security analysis.

Defence: Independent verification of all NIST parameter choices by multiple nations' cryptographic agencies (ANSSI, BSI, GCHQ). This has happened Рђћ and no evidence of backdooring has been found. But healthy scepticism is appropriate.

Cryptographic Agility Рђћ The Most Important Property

Architecture ┬и Defence

Cryptographic agility means designing systems so that the algorithm can be replaced without redesigning the entire system. SIKE proved that no algorithm is guaranteed to survive Рђћ systems must be able to pivot quickly.

  • Key material labelled with algorithm version Рђћ old keys can coexist with new keys during migration
  • Regular algorithm review Рђћ schedule annual review of all cryptographic primitives in use

The Session 3 Core Takeaway

Final Principle

PQC is necessary but not sufficient

Deploying ML-KEM and ML-DSA eliminates the quantum threat to the math layer. It does nothing to fix timing vulnerabilities, fault injection targets, protocol design flaws, or weak RNG. A system using ML-KEM with a timing-vulnerable FO implementation is weaker than a system using ECDH with a constant-time implementation.

Security is only as strong as the weakest layer. PQC deployment must be accompanied by implementation review, hardware hardening for physical-access threat models, protocol enforcement, and operational security Рђћ not just an algorithm swap.

The three questions for every PQC deployment

  • Is the math right? Рђћ Use NIST-standardised algorithms (ML-KEM-768+, ML-DSA-65+). Hybrid mode during transition. Cryptographic agility built in.
  • Is the code right? Рђћ Constant-time implementations, formally verified where possible. Masking for hardware deployments. FO transform integrity verified in code review.
  • Are the people and processes right? Рђћ Hardware RNG verified. Key management updated for larger key sizes. HNDL migration started. Agility to pivot if an algorithm breaks.

Three things a VPN provides

  • Confidentiality Рђћ traffic between you and VPN server is encrypted. ISP sees only encrypted packets to the VPN server.
  • Integrity Рђћ tampered packets are detected and dropped via HMAC or AEAD tag.
  • IP masking Рђћ destination sees VPN server's IP, not yours. Geographic IP location is the VPN server's location.

A VPN does NOT provide: anonymity from the VPN provider, protection from malware on your device, or security after the traffic leaves the VPN server unencrypted.

Why each version was broken

SSL 2.0 Рђћ weak export ciphers mandated by US government + MAC applied to plaintext not ciphertext (MAC-then-encrypt flaw). SSL 3.0 Рђћ POODLE: attacker forces fallback to SSL 3.0, exploits CBC padding oracle. TLS 1.0/1.1 Рђћ BEAST: predictable CBC IV allows chosen-plaintext attack. TLS 1.2 Рђћ mostly safe with correct config, but allowed weak cipher suites (RC4, 3DES, export ciphers, RSA key exchange without forward secrecy). TLS 1.3 Рђћ eliminated all of this by removing weak algorithms entirely, not just making them optional.

What TLS 1.3 Removed Рђћ And Why

Deliberate Removals

Removed cipher suites:

  • RC4 Рђћ statistical biases, broken
  • 3DES Рђћ SWEET32 birthday attack
  • AES-CBC Рђћ padding oracle attacks
  • Export cipher suites Рђћ FREAK, LOGJAM
  • NULL cipher Рђћ no encryption at all
  • MD5 and SHA-1 based MACs Рђћ collision attacks

Removed key exchange mechanisms:

  • RSA key exchange Рђћ no forward secrecy, one key compromise decrypts all past sessions
  • Static DH Рђћ no forward secrecy
  • Anonymous DH/ECDH Рђћ no authentication, trivial MitM
  • EXPORT key exchange Рђћ artificially weakened
  • SRP Рђћ complex, rarely used
  • DSA Рђћ replaced by ECDSA/EdDSA

The key insight behind TLS 1.3 design

TLS 1.2 allowed weak algorithms and relied on correct configuration to exclude them. In practice, misconfigured servers left weak options enabled. TLS 1.3 solved this by removing weak algorithms from the protocol entirely Рђћ there is no configuration option to re-enable them. You cannot accidentally enable RC4 in TLS 1.3 because it does not exist in TLS 1.3.

TLS 1.3 Key Schedule Рђћ Full Derivation

HKDF Key Derivation

TLS 1.3 uses HKDF (HMAC-based Key Derivation Function) to derive all keys from the ECDHE shared secret. The key schedule is a carefully ordered sequence that produces independent keys for every purpose.

Why the key schedule is designed this way

Every derived key is bound to the full handshake transcript hash H. This means keys are different for every session Рђћ even if the same ECDHE keys were reused (which they are not). The hierarchical structure means compromising one derived key does not reveal others. Application keys (used for your HTTPS data) are derived after the full handshake is verified Рђћ ensuring no key is used if the handshake was tampered with.

Why Finished blocks downgrade attacks

The Finished HMAC covers the entire handshake transcript Рђћ including the ClientHello that listed cipher suites. Resumption allows subsequent connections to skip the full handshake. 0-RTT goes further Рђћ sending application data with the very first packet, before any server response. Both have important security tradeoffs.

Why PSK still uses ECDHE

psk_dhe_ke mode (not psk_ke) requires a fresh ECDHE key exchange alongside the PSK. This maintains forward secrecy Рђћ compromising the PSK does not reveal past sessions because each session also used an independent ephemeral ECDHE secret.

psk_ke (PSK without ECDHE) is also defined but loses forward secrecy Рђћ avoid it except for constrained IoT where the compute cost of ECDHE is prohibitive.

Benefit of resumption: Skips certificate validation (expensive chain verification + OCSP check) Рєњ 40-60% CPU reduction on the server for repeat visitors.

Heartbleed Рђћ CVE-2014-0160

Not a cryptographic break Рђћ a buffer over-read in OpenSSL's TLS heartbeat extension. Exposed private keys, passwords, and session tokens from ~500,000 HTTPS servers simultaneously. Discovered April 2014.

What Happened Рђћ The Exact Vulnerability

Memory Safety ┬и Code Layer
# TLS Heartbeat Рђћ RFC 6520 # Purpose: keep-alive ping Рђћ "are you still there?"   • Server private keys (SSL/TLS private key Рђћ could impersonate server forever)
  • Session tokens (hijack active logged-in sessions)
  • Username/password pairs (from recent logins in memory)
  • Secret keys for symmetric encryption (decrypt past/future sessions)

  • Yahoo, LastPass, Imgur, OKCupid confirmed compromised

  • Technical fix: Bounds check Рђћ verify payload_length РЅц actual received data length before memcpy. One line of code. The fix took 2 hours to write and test.
  • Process fix: Memory-safe languages (Rust, Go) make this class of bug impossible. OpenSSL written entirely in C with manual memory management.
  • Detection: Fuzzing would have found this Рђћ libFuzzer on the heartbeat path would have triggered the over-read within seconds.
  • Funding: OpenSSL had ~$2,000/year in donations when Heartbleed hit. The Core Infrastructure Initiative was founded immediately after Рђћ $3.6M/year to critical open-source security libraries.

POODLE & BEAST Рђћ CBC Padding Oracle Attacks

Two related attacks on CBC mode in SSL/TLS. BEAST (2011) Рђћ chosen plaintext on TLS 1.0. POODLE (2014) Рђћ padding oracle on SSL 3.0 fallback. Both exploited the fundamental flaw in how CBC mode was used in SSL/TLS.

CBC Padding Oracle Рђћ The Root Cause

Cryptographic Flaw ┬и Math + Protocol
# CBC decryption P[i] = Decrypt(C[i]) XOR C[i-1] # PKCS#7 padding Рђћ last block must be padded # to block boundary with padding bytes = padding length ... | data | 03 03 03 | Рєљ 3 bytes of padding ... | data | 05 05 05 05 05 | Рєљ 5 bytes # Server decrypts, checks padding validity # If padding invalid Рєњ "padding error" response # If padding valid Рєњ "MAC error" response (different!) # This difference is the oracle # Attacker can distinguish valid from invalid padding # via timing, error message, or connection close behavior

The Attack Mechanism

By modifying the previous ciphertext block (C[i-1]) and observing whether decryption produces valid padding, an attacker can learn the plaintext one byte at a time Рђћ 256 attempts per byte maximum.

For a 16-byte cookie: 16 ├Ќ 128 = ~2,048 requests on average. Automated tools recover session cookies in minutes. POODLE forced SSLv3 fallback (every browser supported SSLv3 for backward compatibility) to exploit this.

  • Disable all SSLv3 Рђћ TLS_FALLBACK_SCSV prevents deliberate downgrade to SSLv3.
  • TLS 1.3 Рђћ removed CBC entirely. All cipher suites are AEAD. POODLE is architecturally impossible in TLS 1.3.

ROBOT Рђћ Return Of Bleichenbacher's Oracle Threat

A 1998 attack on RSA PKCS#1 v1.5 encryption Рђћ resurfaced in 2017 and found on major production servers including Facebook, PayPal, Cisco, and dozens of others. The same 19-year-old vulnerability, still deployed.

Bleichenbacher's Attack Рђћ RSA Padding Oracle

Math + Protocol ┬и 1998/2017
# RSA PKCS#1 v1.5 encryption format EM = 0x00 || 0x02 || PS || 0x00 || M # PS = random non-zero padding (РЅЦ8 bytes) # M = actual message (pre-master secret) # Server decrypts RSA ciphertext, checks: # Does EM start with 0x00 0x02? # If YES: "conformant" (implicit signal) # If NO: "non-conformant" (different handling) # Bleichenbacher 1998: use this as oracle # Multiply ciphertext by s: C' = C ├Ќ sрхЅ mod n # Server decrypts C': is it conformant? # Binary search on M using conformance signal # After ~1 million queries: recover M completely # M = TLS pre-master secret = decrypt all past sessions

Why it resurfaced in 2017

Many implementations fixed the 1998 attack but introduced subtle timing differences Рђћ fast path for conformant, slow path for non-conformant. The oracle was still there, just quieter. B├Хck, Somorovsky, and Young found it on: F5 BIG-IP, Cisco ACE, Citrix NetScaler, Radware, and others. Facebook's main domain was vulnerable. PayPal was vulnerable.

19 years. Same vulnerability. Still deployed on major production infrastructure.

  • RSA-OAEP for encryption Рђћ if RSA encryption is needed (not key exchange), use OAEP padding not PKCS#1 v1.5.
  • TLS 1.3 Рђћ removed RSA key exchange entirely. ROBOT is architecturally impossible in TLS 1.3.
  • Constant-time RSA decryption Рђћ if PKCS#1 v1.5 must be used (legacy), the conformance check must take identical time regardless of result.

The Attack Рђћ Two PDFs, One Hash

Cryptanalysis ┬и Math Layer
# SHAttered result SHA1(PDF_1) = 38762cf7f55934b34d179ae6a4c80cadccbb7f0a SHA1(PDF_2) = 38762cf7f55934b34d179ae6a4c80cadccbb7f0a # PDF_1 and PDF_2 are different files # Same SHA-1 hash Рђћ collision found # Attack complexity: 9.2 ├Ќ 10^18 SHA-1 compressions 110 GPU-years total computation Cost estimate: ~$110,000 (2017 GPU prices) or $45,000 on AWS spot instances # Theoretical birthday bound: 2^80 operations # Actual attack: 2^63.1 Рђћ differential cryptanalysis # 100,000├Ќ cheaper than birthday attack # Structural weakness exploited, not brute force

The Meta-Pattern

Core Insight

Attacks almost never break the math

Of all the case studies in this session and Session 3, only SHAttered and SIKE broke the underlying mathematical assumption. Every other attack Рђћ Heartbleed, POODLE, ROBOT, LOGJAM, FREAK, DigiNotar, Juniper, Equifax Рђћ was a failure of implementation, protocol design, operations, or policy.

This means: switching to PQC algorithms solves the quantum math threat. It does not solve any of these other eleven patterns. All eleven will still exist and still be exploited after PQC migration.

The cost of delayed deprecation

SHA-1 weakness known 2005 Рєњ SHAttered 2017 Рєњ 12 years of exposure. SSL 3.0 flawed since inception Рєњ POODLE 2014 Рєњ nearly 20 years deployed. Export ciphers mandated 1990s Рєњ FREAK/LOGJAM 2015 Рєњ 15+ years after they should have been removed.

The pattern: industry knows an algorithm is weak, delays migration for "compatibility," pays the price when the attack becomes practical. The correct model: deprecate proactively, migrate before attacks are practical. TLS 1.3 embodies this Рђћ removed weak algorithms before breaks occurred, not after.

HSM Рђћ Hardware Security Module

The Gold Standard

An HSM is a physical device that generates, stores, and uses cryptographic keys Рђћ the private key never leaves the HSM. Even with full server compromise, the private key cannot be extracted.

# HSM key operations # Keys generated inside HSM Рђћ never exported hsm.generate_key(algorithm="ML-DSA-65", id="signing-key-v3") # Sign operation Рђћ private key stays in HSM signature = hsm.sign( key_id="signing-key-v3", data=message_hash ) # HSM signs internally, returns signature only # Private key never exits hardware boundary # Physical tamper protection: # Tamper-evident: logs if case opened # Tamper-resistant: zeroises keys if tampered # FIPS 140-3 Level 3: physical attack resistance # PQC HSM challenge: # ML-DSA private key = 4,032 bytes # Many HSMs designed for RSA (256 bytes) # HSM firmware updates needed for PQC key sizes

HSM options by tier

  • Cloud KMS (AWS KMS, Azure Key Vault, GCP KMS) Рђћ HSM-backed, API-driven, auto-scaling. Best for most applications. ~$1/key/month.
  • Dedicated cloud HSM (AWS CloudHSM, Azure Dedicated HSM) Рђћ dedicated hardware, you control the HSM partition. ~$1.50/hour. Required for FIPS 140-3 Level 3.
  • On-premises HSM (Thales, Utimaco, nCipher) Рђћ physical device in your datacenter. $20KРђЊ$100K+. Required when data cannot leave premises.
  • Software KMS (HashiCorp Vault) Рђћ keys in memory/encrypted storage. No physical tamper protection. Good for secrets management, not long-term CA keys.

Service meshes (Istio, Linkerd) implement mTLS transparently Рђћ inject sidecar proxies, handle cert rotation automatically, enforce policy. Applications see plaintext; the mesh handles all crypto. This is the cleanest way to deploy at scale.

Master Principle

What security actually requires

Cryptography gives you the tools. Key management determines whether you use them correctly. Operations determines whether they stay correct under pressure. People determine whether the whole system holds under social engineering, insider threats, and the chaos of real production.

A system using AES-128-GCM with proper key management, constant-time implementation, and disciplined operations is more secure than a system using AES-256-GCM with keys in config files, no rotation, and no monitoring. The algorithm is the smallest part of real-world security.
