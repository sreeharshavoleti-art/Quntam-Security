Section 01

Symmetric Cryptography — How It Works

One shared secret key encrypts and decrypts. The math transforms plaintext through multiple rounds so thoroughly that reversing it without the key is computationally impossible.

AES — How Each Round Works

FIPS 197 · Block Cipher

AES operates on a 4×4 grid of bytes called the state. Every round applies four transformations in sequence. AES-128 does 10 rounds; AES-256 does 14 rounds.

SubBytes — Confusion

Every byte in the 4×4 state is replaced by a fixed lookup table called the S-Box. This substitution is non-linear — it makes the relationship between key and ciphertext impossible to express as a simple equation.

# S-Box substitution (simplified) state[r][c] = SBOX[ state[r][c] ] # each byte → new byte # e.g. 0x53 → 0xED (from fixed S-Box table) # Non-linear — breaks algebraic attacks
ShiftRows — Diffusion (horizontal)

Each row of the 4×4 state is shifted left by 0,1,2,3 positions respectively. Bytes from one column spread into different columns — so one changed byte affects the whole block.

# Row 0: no shift [a0 a1 a2 a3] # Row 1: shift left 1 [b1 b2 b3 b0] # Row 2: shift left 2 [c2 c3 c0 c1] # Row 3: shift left 3 [d3 d0 d1 d2]
MixColumns — Diffusion (vertical)

Each column of 4 bytes is multiplied by a fixed matrix in GF(2⁸) — the Galois Field. One changed byte now affects all 4 bytes in the column. Combined with ShiftRows, changing 1 input byte affects all 16 output bytes after 2 rounds (avalanche effect).

# Matrix multiplication in GF(2^8) [s'0] [2 3 1 1] [s0] [s'1] = [1 2 3 1] [s1] (mod irreducible polynomial) [s'2] [1 1 2 3] [s2] [s'3] [3 1 1 2] [s3]
AddRoundKey — Key Injection

The 128-bit round key (derived from the master key via Key Schedule) is XORed with the entire state. XOR is reversible only if you know the key. This is the only step that uses the secret key directly.

# XOR each byte of state with round key state[r][c] = state[r][c] XOR roundKey[r][c] # Repeat all 4 steps for 10/12/14 rounds # Final round: SubBytes + ShiftRows + AddRoundKey (no MixColumns)

Section 02

Asymmetric Cryptography — How It Works

Public-private key pairs. The mathematical relationship makes one direction easy and the reverse computationally infeasible — that asymmetry is the security.

RSA — Step by Step Mechanism

Integer Factorisation
RSA Key Generation
# Step 1: Pick two large random primes p = 61, q = 53 # (tiny example — real: 1024-bit primes) # Step 2: Compute modulus n = p × q = 3233 # public modulus # Step 3: Euler's totient φ(n) = (p-1)(q-1) = 3120 # Step 4: Choose public exponent e = 17 # must be coprime with φ(n) # Step 5: Compute private exponent d = e⁻¹ mod φ(n) = 2753 # modular inverse # Public key: (n=3233, e=17) # Private key: (n=3233, d=2753)
RSA Encryption & Decryption
# Encrypt message M=65 with public key (e=17, n=3233) C = M^e mod n C = 65^17 mod 3233 = 2790 # ciphertext # Decrypt ciphertext C=2790 with private key (d=2753, n=3233) M = C^d mod n M = 2790^2753 mod 3233 = 65 # original message recovered # Real RSA: add OAEP padding before encrypting # Real RSA: M and C are thousands of bits long
Why RSA Is Hard (and why quantum breaks it)

To find d, an attacker needs φ(n) = (p-1)(q-1). To get that, they need p and q. But p and q are secret — and finding them means factoring n (a huge number). Classically infeasible. But Shor's algorithm on a quantum computer finds the period of a modular exponential function, which directly yields the factors of n — breaking RSA completely.

ECC — Elliptic Curve Mechanism

ECDLP Hardness

Points on an elliptic curve y² = x³ + ax + b form a group. "Adding" two points follows a geometric rule — draw a line through them, find where it hits the curve, reflect.

# Point addition: P + Q = R # Draw line through P and Q # Find third intersection point # Reflect across x-axis → R # Scalar multiplication: Q = k × G # add G to itself k times # k = private key (secret) # G = generator point (public) # Q = public key (public)

The Hard Problem

Given Q and G, find k. This is the Elliptic Curve Discrete Logarithm Problem (ECDLP). Going forward (k×G) is fast. Going backward (Q→k) is computationally infeasible classically. Quantum Shor's solves it.

Hash Functions — How They Work

One-way compression. Data of any size → fixed-size digest. The internal structure ensures you cannot reverse the process or find two inputs with the same output.

HMAC — Authentication Mechanism

Integrity + Authenticity
# HMAC = keyed hash — proves both integrity AND authenticity ipad = 0x36 repeated 64 times opad = 0x5C repeated 64 times HMAC(K, M) = H( (K ⊕ opad) ∥ H( (K ⊕ ipad) ∥ M ) ) # Two-layer hash prevents length extension attacks # Without key K, attacker cannot forge valid HMAC

Without the secret key K, even knowing the hash algorithm, an attacker cannot produce a valid HMAC for a tampered message. This is why HMAC proves authenticity — not just integrity.

Section 04

Post-Quantum Cryptography — How It Works

New math problems that neither classical nor quantum computers can solve efficiently. LWE (Learning With Errors) is the core foundation of ML-KEM and ML-DSA.

The LWE Problem — Foundation of All Lattice PQC

Learning With Errors
Step 1 — Normal Linear System (solvable)
# Simple system — easy to solve with Gaussian elimination A × s ≡ b (mod q) # Example: 1000 equations, 1000 unknowns, no errors # Classical: solvable in polynomial time # Quantum: also solvable easily
Step 2 — Add Small Random Noise (impossible to reverse)
# Add tiny random error e to every equation output A × s + e ≡ b (mod q) # e is drawn from a narrow Gaussian distribution # e.g., e values are typically -2, -1, 0, 1, 2 # Now the system is INCONSISTENT # No exact solution exists # Gaussian elimination doesn't work # Brute force: 2^256 possibilities for s # Classical: infeasible. Quantum: also infeasible.
Analogy: Imagine 1000 equations written on a whiteboard but someone smudged each answer by ±2. You can see all the equations. The smudge is tiny — but it makes it impossible to find the original answers. Even if you had a quantum computer.
Step 3 — Why Quantum Cannot Solve LWE
  • Shor's algorithm works by finding periodicity. LWE has no mathematical period — the noise destroys all structure Shor needs.
  • Grover's algorithm can search in √N time — but for LWE with n=768, the search space is 2^768. Grover reduces this to 2^384 — still astronomically infeasible.
  • BKZ lattice reduction is the best known classical lattice attack — but it runs in exponential time in the security parameter. ML-KEM-768 is designed so BKZ would take longer than the age of the universe.

Pre-Session 3 · Foundation 01

Computational Complexity

When cryptographers say an algorithm is "secure" — they mean breaking it requires an infeasible number of operations. You need to understand exactly what "infeasible" means mathematically.

P vs NP — The Core Idea

Complexity Theory

Every computational problem falls into a complexity class — a measure of how hard it is to solve as the input size grows. Cryptography lives at the boundary between "easy" and "hard".

ClassMeaningExampleCrypto relevance
PSolvable in polynomial time — fastSorting a list, adding numbersEncryption/decryption must be in P — otherwise it's too slow to use
NPSolution verifiable in polynomial time — answer easy to check, hard to findSudoku, jigsaw puzzleBreaking crypto must be NP or harder — otherwise anyone can crack it
NP-HardAt least as hard as the hardest NP problemsTravelling salesman, satisfiabilitySyndrome decoding (code-based PQC) is NP-Hard
ExponentialTime grows as 2ⁿ — doubles with every extra bitBrute-force key searchAES-256 brute force: 2²⁵⁶ — more operations than atoms in universe
Analogy: Locking a padlock is P (fast — click, done). Finding the combination by trying every number is exponential (slow — billions of tries). Cryptography is designed so that encryption is P and breaking it is exponential.

What "2¹²⁸ Operations" Actually Means

Real Numbers

Security levels are expressed as bit security — the number of operations needed to break the algorithm is 2ⁿ where n is the security level.

Security LevelOperations neededReal-world time (fastest computer)Context
2⁴⁰~1 trillionSeconds to minutesBroken — don't use
2⁵⁶~72 quadrillionHours (DES cracked 1999)Broken
2⁸⁰~1.2 × 10²⁴DecadesMarginal — avoid

Why This Matters for PQC

Grover's algorithm halves security in bit terms. AES-128 (2¹²⁸) becomes 2⁶⁴ effective under quantum — still 18 quintillion operations but borderline. AES-256 (2²⁵⁶) becomes 2¹²⁸ — still completely safe. This is why we recommend AES-256 for the quantum era.

Polynomial vs Exponential — Why the Gap is Everything

Growth Rates

Polynomial growth (safe operations):

# Doubling input size doubles work n=128 → 128 operations (linear) n=256 → 256 operations n=512 → 512 operations # Or squares it (quadratic) n=128 → 16,384 ops n=256 → 65,536 ops # still manageable

Exponential growth (attack side):

# Adding 1 bit DOUBLES the work n=128 → 2¹²⁸ operations n=129 → 2¹²⁹ operations # 2× harder n=256 → 2²⁵⁶ operations # incomprehensibly more # This gap is why crypto works # Encrypt: polynomial. Break: exponential.

Sub-exponential attacks (like the Number Field Sieve for factoring RSA) are between polynomial and exponential — faster than brute force but still impractical at large key sizes. This is why RSA needs 2048+ bits while AES only needs 128 bits for equivalent security.

Security Reductions — How We Know PQC Is Hard

Proof Technique

Cryptographers prove security by reduction: if you can break algorithm X, then you can solve hard problem Y. Since Y is assumed hard, X must also be hard.

If attacker breaks ML-KEM
→
Then attacker can solve MLWE
→
But MLWE assumed hard
→
Therefore ML-KEM is hard

This is why SIKE's break was so significant — it broke the underlying hard problem assumption (SIDH), not just a specific implementation. The entire security foundation collapsed instantly.

Pre-Session 3 · Foundation 02

Probability & The Birthday Paradox

Collision attacks, Grover's quantum speedup, and the 2× rule for hash security — all come from one piece of probability theory. Understand this and attack numbers become intuitive.

The Birthday Paradox — Why It Matters

Probability Foundation

How many people do you need in a room before two share a birthday? Intuition says ~183 (half of 365). Reality: just 23 people gives 50% probability. This counterintuitive result destroys naive hash security assumptions.

# Birthday problem formula P(collision) ≈ 1 - e^(-n²/2N) # n = people in room # N = total possible birthdays (365) # 50% collision at n ≈ √N n ≈ √365 ≈ 19.1 # Reality: 23 people → 50.7% chance # For hash with 2^b output bits: # 50% collision at ~2^(b/2) attempts # SHA-256 (256-bit): 2^128 attempts # NOT 2^256 as you might expect!

Why √N not N?

You don't need to find a specific collision — any two matching values count. Every new item is compared against ALL previous items. With n items, there are n(n-1)/2 ≈ n²/2 pairs being compared. When n²/2 ≈ N, expect a collision. So n ≈ √N.

This is why hash functions need 2× the bit length of their claimed security: 128-bit security needs 256-bit hash output.

Birthday Attack on Hash Functions

How MD5/SHA-1 Were Broken

The birthday paradox is the mathematical foundation of collision attacks on hash functions. You don't need to find a specific input — you just need any two inputs that produce the same hash.

Grover's Algorithm — Quantum Birthday Effect

Quantum Speedup

Grover's algorithm gives a quadratic speedup for unstructured search problems — exactly the same √N relationship as the birthday paradox, but applied to quantum searching.

# Classical brute-force key search AES-128: 2¹²⁸ attempts needed AES-256: 2²⁵⁶ attempts needed # Grover's quantum search # Searches N items in √N time AES-128: √(2¹²⁸) = 2⁶⁴ quantum ops AES-256: √(2²⁵⁶) = 2¹²⁸ quantum ops # Effective security under quantum: AES-128 → 64-bit ← borderline AES-256 → 128-bit ← still safe

Why Grover doesn't break PQC symmetric

AES-256 with 128-bit quantum-effective security is still completely safe — 2¹²⁸ quantum operations is infeasible. This is why we don't need to replace AES — we just make sure we're using AES-256 not AES-128.

SHA-256: pre-image resistance drops from 2²⁵⁶ to 2¹²⁸ under Grover — still safe. SHA-3-256: same. This is why symmetric crypto and hash functions survive the quantum era.

Pre-Session 3 · Foundation 03

Lattice Geometry

ML-KEM and ML-DSA are built on lattice problems. To understand how they get attacked (BKZ, LLL), you need to understand what a lattice actually is geometrically — not just algebraically.

What Is a Lattice?

Geometric Foundation

A lattice is an infinite regular grid of points in n-dimensional space. It is defined by a basis — a set of vectors from which every lattice point can be reached by integer combinations.

# 2D lattice defined by two basis vectors b₁ = [3, 0] # first basis vector b₂ = [1, 2] # second basis vector # Every lattice point = integer combination P = a·b₁ + b·b₂ where a,b ∈ ℤ # Examples: [0,0] = 0·b₁ + 0·b₂ (origin) [3,0] = 1·b₁ + 0·b₂ [4,2] = 1·b₁ + 1·b₂ [7,2] = 2·b₁ + 1·b₂ [2,4] = 0·b₁ + 2·b₂ # In crypto: n = 256, 512, 768 dimensions # Same idea — incomprehensibly larger space

Key insight: same lattice, different bases

The same lattice can be described by many different bases. A "good" basis has short, nearly-orthogonal vectors. A "bad" basis has long, nearly-parallel vectors.

Good basis → easy to navigate the lattice, solve problems efficiently. This is what the private key holder has.

Bad basis → hard to navigate, hard to solve problems. This is what an attacker sees (the public key).

Lattice cryptography security = the gap between a good basis and a bad basis.

The Three Hard Lattice Problems

Security Foundations
SVP — Shortest Vector Problem

Given a lattice basis, find the shortest non-zero vector in the lattice. In 2D this is trivial — you can see it visually. In 1000 dimensions it is computationally infeasible.

# 2D: trivial — just look at the grid # 100D: hard — best algorithms take exponential time # 1000D: infeasible — this is the PQC security basis # Best known SVP algorithm: BKZ (Block Korkine-Zolotarev) # BKZ complexity: 2^(0.292n) — exponential in dimension n # For n=768: 2^224 operations — infeasible

SVP hardness is the foundation of lattice cryptography. ML-KEM security reduces to an approximate version of SVP — if you could solve SVP efficiently, you could break ML-KEM.

CVP — Closest Vector Problem

Given a lattice and a target point (which may NOT be on the lattice), find the lattice point closest to the target. Even harder than SVP in general.

# Target point t (not on lattice) # Find lattice point v minimising ∥t - v∥ # Directly related to LWE decoding: # b = As + e ← b is close to As (on lattice) # Finding s = finding closest lattice point to b # = CVP — computationally infeasible without s

ML-KEM decapsulation is essentially solving an easy CVP instance — easy because the private key s gives you the good basis to navigate directly to the answer. An attacker without s faces the hard CVP.

LWE — Learning With Errors (Lattice View)

LWE is equivalent to solving a noisy CVP problem. The vector b = As + e is a lattice point (As) with a small error (e) added. Finding s means finding the lattice point closest to b — which is CVP.

# LWE instance = CVP instance Lattice L = { As : s ∈ ℤqⁿ } # lattice spanned by columns of A Target point = b = As + e # b is NEAR lattice but not on it # Solve LWE = find closest lattice point to b # = solve CVP in L # = exponentially hard for large n # Private key holder: knows s directly → trivial # Attacker: must solve CVP → infeasible

Basis Reduction — What LLL and BKZ Do

Attack Foundation — Session 3 Preview

Lattice basis reduction algorithms take a bad basis and try to find a better (shorter, more orthogonal) basis for the same lattice. A sufficiently good basis can solve SVP/CVP — breaking the crypto.

LLL Algorithm (1982)

Lenstra-Lenstra-Lovász. Runs in polynomial time. Finds a "somewhat" reduced basis — not the shortest vector, but close. Good enough to break low-dimension lattices and small parameters.

Used to break RSA with small exponents, factoring via lattices, and early lattice crypto with weak parameters.

BKZ Algorithm (1987–2011)

Block Korkine-Zolotarev. Stronger than LLL — applies SVP oracle to blocks of dimensions β. Quality improves with block size β but cost grows exponentially in β.

Best known attack on LWE-based PQC. ML-KEM-768 parameters chosen so BKZ with any practical β requires 2¹⁸⁵+ operations — infeasible.

# BKZ complexity vs block size β β=20 → polynomial → breaks toy lattices β=50 → 2^20 ops → breaks weak parameters β=100 → 2^50 ops → significant effort β=200 → 2^100 ops → infeasible classically β=400 → 2^200 ops → ML-KEM-768 security level # ML-KEM-768 requires BKZ-β≈830 to break # Cost: 2^(0.292×830) ≈ 2^242 operations # Completely infeasible — classical or quantum
⚠️ As computers get faster and BKZ implementations improve, security margins get tighter. This is why NIST chose parameters with significant security margin above the minimum — room to absorb algorithmic improvements over decades.

Why High Dimensions = Security

The Core Intuition

In low dimensions, geometry is navigable — you can visualise and solve problems. As dimensions grow, space becomes incomprehensibly vast and our geometric intuitions completely break down.

# Volume of unit sphere by dimension 1D: 2 2D: π ≈ 3.14 3D: 4π/3 ≈ 4.19 10D: π⁵/120 ≈ 2.55 100D: ≈ 10⁻⁴⁰ ← nearly zero! 1000D: essentially 0 # High-dim sphere has almost no volume # All mass is in a thin shell near surface # Geometric intuition = completely useless

Curse of Dimensionality

In high dimensions, every pair of points is roughly the same distance apart. There are no obvious "short" vectors standing out. Finding the shortest vector is like finding a needle in a haystack where all straws look the same length.

ML-KEM uses n=256 polynomial coefficients over a module of rank k=3 → effectively ~768 dimensional lattice. Completely opaque to any known algorithm.

Pre-Session 3 · Foundation 04

Quantum Computing Basics

You don't need to build a quantum computer — but you need to understand why Shor's algorithm breaks RSA, why it fails on LWE, and what makes quantum speedup possible at all.

Qubits — What They Actually Are

Quantum Foundation

A classical bit is 0 or 1. A qubit can be in a superposition — a combination of 0 and 1 simultaneously — until it is measured. Measurement collapses it to exactly 0 or 1.

# Classical bit bit = 0 OR bit = 1 # one at a time # Qubit state (Dirac notation) |ψ⟩ = α|0⟩ + β|1⟩ # α, β are complex amplitudes # |α|² + |β|² = 1 (probabilities sum to 1) # Measure → get 0 with prob |α|², 1 with prob |β|² # n qubits = superposition of 2ⁿ states # 3 qubits: |000⟩,|001⟩,...,|111⟩ all at once # 300 qubits: 2³⁰⁰ states — more than atoms in universe

Why superposition is NOT "try all at once"

A common misconception: quantum computers don't simply "try all possibilities simultaneously." When you measure, you get ONE result. The trick is using quantum algorithms to amplify the probability of the correct answer before measuring.

Quantum algorithms work by constructing interference patterns — correct answers interfere constructively (amplitudes add), wrong answers interfere destructively (amplitudes cancel). Only Shor's, Grover's, and a few others achieve this elegantly.

Quantum Interference — The Core Mechanism

Why Quantum Works

Quantum speedup comes from interference — the same phenomenon that creates light/dark bands when light passes through two slits. Correct answers get amplified; wrong answers cancel out.

Analogy: Imagine dropping a pebble in a pond — ripples spread in all directions. Now drop two pebbles simultaneously at different spots. Where ripples meet in phase — they amplify (constructive interference). Where they meet out of phase — they cancel (destructive interference). Quantum algorithms engineer this so the answer's amplitude grows and all wrong answers' amplitudes shrink to zero.
# Quantum algorithm structure 1. Initialise qubits in superposition of ALL possible inputs 2. Apply quantum gates that encode the problem 3. Use interference to amplify correct answer's amplitude 4. Measure → get correct answer with high probability # The art is in step 3 — designing the interference pattern # Only works for problems with exploitable mathematical structure # LWE has no such structure → quantum interference cannot help

Shor's Algorithm — Why It Breaks RSA

The Quantum Threat
Step 1 — The Key Insight: Period Finding

Factoring n = p×q is hard classically. But Shor showed that factoring REDUCES to finding the period of a function — and period finding is something quantum computers can do exponentially faster using the Quantum Fourier Transform.

# Shor's algorithm (high level) # Goal: factor n = p × q 1. Pick random a (1 < a < n) 2. Find period r of f(x) = aˣ mod n # f(0)=1, f(1)=a, f(2)=a², ... f(r)=1 again # Classical: exponential time to find r # Quantum (QFT): polynomial time! 3. From r: compute gcd(aʳ/² ± 1, n) # High probability → p or q revealed # RSA private key derivable from p, q
Step 2 — Quantum Fourier Transform

The classical Discrete Fourier Transform (DFT) takes O(n log n) time. The Quantum Fourier Transform (QFT) takes O((log n)²) — exponentially faster. It reveals periodic structure in quantum superposition states.

# QFT maps amplitude patterns to frequency patterns # If function has period r, QFT produces peaks at multiples of 1/r Classical DFT on N points: O(N log N) Quantum QFT on N points: O((log N)²) ← exponential speedup # For RSA-2048: N = 2^2048 Classical: O(2^2048 × 2048) ← infeasible Quantum: O(2048²) = O(4M) ← polynomial, trivial # This is why Shor's destroys RSA and ECC # Both rely on problems with hidden periodic structure
Step 3 — Why Shor's Completely Fails on LWE

Shor's algorithm needs a hidden periodic structure to exploit. LWE has no such structure — the random noise completely destroys any periodicity. QFT on a noisy LWE instance produces a flat, random amplitude distribution — no peaks, no information.

# RSA: f(x) = aˣ mod n has clear period r # QFT → sharp peaks at multiples of 1/r # Shor extracts r → factors n → breaks RSA ✓ # LWE: b = As + e (random noise added) # No periodic structure in b # QFT → flat random distribution # No peaks → no information → Shor extracts nothing # LWE completely immune to Shor's algorithm ✓ # This is THE reason lattice PQC was chosen: # LWE noise destroys the structure Shor needs

Grover's Algorithm — The Other Quantum Threat

Quadratic Speedup

While Shor's gives exponential speedup for structured problems, Grover's gives quadratic speedup for ANY unstructured search. It is weaker but universal — applies to brute-forcing keys, searching databases, and more.

# Grover's algorithm # Search N unsorted items for one marked item Classical: O(N) steps — check each one Grover: O(√N) steps — quadratic speedup # AES-256 key search: Classical: 2^256 operations Grover: 2^128 operations ← still infeasible # How: amplitude amplification # Each iteration amplifies target item's amplitude # After √N iterations → measure → target found

Why it's the existential threat

Unlike Grover's which weakens but doesn't destroy, Shor's completely eliminates RSA and ECC security. There is no "double the key size" fix — the algorithm finds the answer in polynomial time regardless of key size. The only fix is replacing the algorithm entirely with PQC.

Grover's Algorithm — Quantum Brute Force

Quantum · Math Layer

Quadratic speedup for any unstructured brute-force search. Effectively halves the bit-security of symmetric ciphers and hash functions. Dangerous but manageable — the fix is doubling key sizes.

# Grover's effect on symmetric crypto AES-128: 2^128 → 2^64 effective ⚠️ borderline AES-192: 2^192 → 2^96 effective ✅ safe AES-256: 2^256 → 2^128 effective ✅ safe # Grover's effect on hash functions SHA-256 pre-image: 2^256 → 2^128 ✅ safe SHA-1 pre-image: 2^160 → 2^80 ⚠️ marginal MD5 pre-image: 2^128 → 2^64 ❌ weak # Fix: AES-256 not AES-128 # Fix: SHA-256+ not SHA-1/MD5

Grover's is NOT catastrophic

Unlike Shor's which destroys RSA, Grover's only halves security in bit terms. AES-256 with 128-bit quantum-effective security is still completely safe — 2^128 quantum operations is physically infeasible.

Hash collision resistance under quantum drops by cube root (BHT algorithm). SHA-384 and SHA-512 provide comfortable margin for the quantum era.

General Number Field Sieve (GNFS)

Classical · Math Layer

The best known classical algorithm for factoring large integers — the mathematical basis of RSA attacks. Sub-exponential complexity: faster than brute force but still infeasible at 2048+ bits. This is why RSA needs large keys even against classical computers.

# GNFS practical results RSA-512: broken 1999 (~2^60 ops) RSA-768: broken 2009 (~2^67 ops) RSA-1024: est. ~2030 (~2^80 ops) RSA-2048: ~2^112 ops ← infeasible classically RSA-4096: ~2^140 ops ← very safe classically # vs Shor's (quantum) RSA-2048: polynomial time ← trivial for quantum

GNFS vs Shor's key difference

GNFS is sub-exponential — manageable by increasing key size (2048-bit RSA safe classically for decades). Shor's is polynomial — doubling RSA key size barely increases Shor's difficulty. No classical key size defeats a quantum computer running Shor's. This is why GNFS is manageable but Shor's requires replacing the algorithm entirely.

Harvest Now, Decrypt Later (HNDL)

Strategic · Long-term Threat

Nation-state adversaries intercept and store encrypted traffic today — even though they cannot decrypt it yet. When quantum computers become capable, all stored data gets decrypted retroactively. This is actively happening right now.

ECDHE does NOT protect against HNDL

Forward secrecy (ECDHE) prevents past session decryption if the long-term key is stolen. But an adversary records the ECDHE handshake itself, later runs Shor's to get the ephemeral private key, then decrypts the session.

Only defence: hybrid X25519 + ML-KEM key exchange today. ML-KEM is immune to Shor's — HNDL cannot decrypt sessions using ML-KEM.

Part 2
Attacks on Post-Quantum Cryptography (PQC)

PQC algorithms are mathematically quantum-resistant — but the code implementing them, the hardware running them, and the protocols using them remain vulnerable.

Side-Channel Attacks (SCA)

Code Layer · Physical

Do not attack the mathematics. Attack the physical implementation — measuring what the device leaks while it computes. Even a mathematically perfect algorithm can be completely broken by a poor implementation.

Power Analysis — SPA and DPA

Measure electrical power a device consumes during cryptographic operations. Power consumption correlates with data being processed — different key bits cause measurably different power patterns.

# Simple Power Analysis (SPA) — RSA example if bit == 1: square_and_multiply() # more power if bit == 0: square_only() # less power # Single power trace → full binary key sequence # Differential Power Analysis (DPA) — statistical # Many traces + correlation with guessed key bits # Works even with noise — statistics win over thousands of traces # PQC vulnerability: ML-KEM Gaussian sampling # Power proportional to sampled secret polynomial values # FALCON floating-point sampler leaks through power traces
Timing Analysis — Branch and Cache Timing

If an operation takes different time depending on secret values, an attacker measuring execution time can extract the secret. Differences as small as nanoseconds are exploitable with enough measurements.

# Non-constant-time comparison (VULNERABLE) def compare(a, b): for i in range(len(a)): if a[i] != b[i]: return False # exits early — time leaks position return True # Constant-time comparison (SAFE) def compare_ct(a, b): result = 0 for i in range(len(a)): result |= a[i] ^ b[i] # always runs all iterations return result == 0 # PQC: ML-KEM NTT polynomial multiplication # Table lookups in NTT can leak via cache timing # Co-located process observes cache hit/miss patterns
Electromagnetic Emission Analysis (EMA)

Every electronic circuit radiates EM signals as it operates. These emissions carry information about data being processed — measurable from a distance without physical contact.

# EMA attack setup EM probe near device (or antenna metres away) ↓ Capture EM signal during PQC operation ↓ Frequency analysis + correlation ↓ Extract secret polynomial coefficients # Real attack (Ruhr University 2021): # Extracted ML-KEM secret keys from microcontroller # Using EM probes — no physical contact # Works even on constant-time implementations # EM leakage harder to eliminate than timing
SCA Countermeasures for PQC
  • Masking — split secret s into random shares: s = s1 + s2. Process each separately. Leaking one share reveals nothing about s.
  • Constant-time code — all branches and memory accesses take identical time regardless of secret values.
  • Shuffling — randomise operation order so traces don't align across measurements.
  • Hardware shielding — Faraday cages, EM shielding in PCB design.
  • Noise injection — random dummy operations mask real power patterns.
  • Formal verification — mathematically prove constant-time property of code.
⚠️ Masking PQC costs 2-5x more computation and memory than unmasked — a major challenge for IoT and smartcard deployments already tight on resources.

Fault Injection Attacks (FIA)

Code Layer · Physical

Intentionally cause computational errors during cryptographic operations. The faulty output compared with the correct output reveals secret key information. A single successful fault can break an otherwise secure implementation.

# Fault injection methods Voltage glitch → CPU skips instruction Clock glitch → double-execute one step Laser pulse → flip specific memory bit EM pulse → corrupt register value Temperature → cause unpredictable errors # DFA on ML-DSA signing 1. Get correct signature S = sign(M, key) 2. Inject fault — skip rejection check 3. Get faulty S' = sign(M, key) with error 4. S - S' = f(key, fault) 5. Repeated faults → full key recovery

FIA on PQC specifically

ML-DSA signing has a rejection step — if output norm is too large, it retries. If a fault forces it to skip this check and output an invalid signature — that signature leaks secret key coefficients.

FALCON is particularly vulnerable — its rejection sampling step, if faulted, can leak the secret NTRU basis directly in a single attack.

Countermeasures: double computation, redundant checks, voltage/clock monitors, physical tamper protection.

Machine Learning-Driven Side-Channel Attacks

AI-Assisted · Code Layer

Neural networks trained on power/EM traces learn the correlation between traces and secret key values automatically — no algorithm knowledge required. Breaks implementations that resist traditional DPA.

# Traditional DPA Attacker manually models P(key | power_trace) Requires knowing algorithm structure Works only if model matches implementation # ML-driven SCA (Deep Learning) Train CNN/MLP on labelled power traces: input: trace (time series values) output: key byte value Model learns correlation automatically No manual modelling needed Works on masked implementations Works on unknown algorithms

Why dangerous for PQC

PQC algorithms have more complex internal structure than AES — making traditional DPA harder to model. Neural networks don't need an explicit model — they learn the power-to-secret mapping from data.

Research (2022-2024) showed deep learning breaking first-order masked ML-KEM — implementations specifically designed to resist traditional DPA. The masking that defeats DPA is insufficient against neural networks.

Decryption Failure Oracle Attacks

Math + Code · PQC Specific

Lattice-based schemes have a tiny but non-zero probability of decryption failure. If an attacker can observe which ciphertexts cause failures, they extract information about the secret key bit by bit.

# ML-KEM failure rates (by design tiny) ML-KEM-512: ~2^-139 per decryption ML-KEM-768: ~2^-164 per decryption ML-KEM-1024: ~2^-174 per decryption # Oracle attack 1. Craft ciphertext c' near decrypt boundary 2. Send to oracle → observe success/fail 3. One bit of key info per query 4. ~2^40 queries → full key recovery # ML-KEM defence: FO Transform # Re-encrypts during decapsulation # Rejects any mismatching ciphertext # Eliminates the oracle entirely # If FO check is faulted → oracle exposed

The FO Transform Protection

ML-KEM applies the Fujisaki-Okamoto transform — making the scheme CCA2-secure. During decapsulation, ML-KEM re-encrypts and rejects any ciphertext that doesn't match exactly — eliminating the decryption oracle.

Critical: If the FO check is bypassed or faulted, the full oracle is exposed. A single missing line of code in the rejection check can compromise the entire key. This is a prime target for fault injection attacks.

Breaking the Math Layer

Cryptanalysis — finding mathematical shortcuts that reduce the work below the security threshold. This is the hardest attack layer and the one PQC is specifically designed to resist.

BKZ Lattice Reduction — The Best Known Attack on LWE

Math Layer · Cryptanalysis

Block Korkine-Zolotarev (BKZ) is the most powerful known algorithm for finding short vectors in lattices — and therefore the best tool for attacking ML-KEM and ML-DSA. Understanding BKZ is understanding the exact hardness assumption PQC relies on.

What BKZ Does — Basis Reduction

Given a bad lattice basis (long, non-orthogonal vectors — what an attacker has from the public key), BKZ tries to find a better basis with shorter, more orthogonal vectors. A sufficiently good basis can solve SVP — finding the shortest vector — which breaks the LWE problem.

# BKZ high-level algorithm Input: bad basis B (from public key — long vectors) Output: better basis B' (shorter vectors) For each block of β consecutive basis vectors: 1. Project out other dimensions 2. Solve SVP in β-dimensional subspace 3. Use short vector found to reduce other vectors 4. Repeat until no more improvement # Larger β = better reduction = shorter vectors found # But: SVP in β dimensions costs 2^(0.292β) # So BKZ-β costs O(poly(n) × 2^(0.292β)) total # Quality vs cost: bigger block = better but slower

The key insight: BKZ doesn't solve LWE directly. It finds a shorter basis, then uses that basis to make LWE easier to solve. If BKZ can find a short enough vector, it can recover the secret s from the noisy equation As + e = b.

Block Size β vs Attack Cost
# BKZ cost model: 2^(0.292β + 16.4) classical ops β=20: 2^22 ops → toy lattices only β=40: 2^28 ops → practical today (academic research) β=60: 2^34 ops → state-of-art implementation β=100: 2^45 ops → years of compute β=200: 2^75 ops → infeasible β=400: 2^133 ops → far infeasible β=830: 2^258 ops → required to break ML-KEM-768 # Quantum BKZ speedup via Grover's # Quantum cost: 2^(0.265β) — modest improvement β=830 quantum: 2^(0.265×830) = 2^220 ops # Still completely infeasible # ML-KEM parameter choices (NIST security levels): ML-KEM-512: requires β≈380 → classical 2^114, quantum 2^101 ML-KEM-768: requires β≈830 → classical 2^242, quantum 2^220 ML-KEM-1024: requires β≈1050→ classical 2^307, quantum 2^278

Algebraic Attacks — Exploiting Structure in Polynomial Rings

Math Layer · Structural

ML-KEM and ML-DSA don't use plain LWE — they use Module LWE (MLWE) over polynomial rings for efficiency. This extra structure is a potential attack surface — structured problems can sometimes be easier than unstructured ones.

# Plain LWE (no structure) A ∈ Z_q^(m×n), s ∈ Z_q^n b = As + e ← random matrix A # Module-LWE (structured ring) A ∈ R_q^(k×k), s ∈ R_q^k b = As + e ← A over ring R_q = Z_q[x]/(x^n+1) # Ring structure = polynomials multiply efficiently # But ring structure = potential algebraic attack surface # Subfield attacks: exploit ring automorphisms # Twisted embedding attacks: use complex geometry

Why Ring-LWE was chosen despite risk

Plain LWE is more conservative but requires much larger keys (megabytes). Ring-LWE enables the compact 1.2KB keys of ML-KEM. After 15+ years of analysis, no algebraic attack has succeeded against properly parameterised Ring-LWE or Module-LWE.

The choice is a calculated tradeoff: accept slightly more structural risk in exchange for practical key sizes. ML-KEM-768's parameters were specifically chosen to resist all known algebraic attacks with significant margin.

The SIKE Collapse — A Complete Case Study

Math Layer · Historical Break

SIKE (Supersingular Isogeny Key Encapsulation) was a NIST PQC finalist broken in 62 minutes on a laptop in 2022. It is the clearest proof that math creates AND breaks security — and the most important warning for all PQC deployments.

What SIKE Was — Isogeny-Based Cryptography

SIKE based its security on the Supersingular Isogeny Diffie-Hellman (SIDH) problem: given two supersingular elliptic curves E and E/⟨P⟩ connected by a secret isogeny (a special curve homomorphism) φ, find φ.

# SIKE key exchange (simplified) Alice: pick secret isogeny φ_A: E → E_A Bob: pick secret isogeny φ_B: E → E_B Alice sends E_A and images φ_A(P_B), φ_A(Q_B) Bob sends E_B and images φ_B(P_A), φ_B(Q_A) # Shared secret: j-invariant of E_AB Alice: compute φ_A(E_B) → E_AB Bob: compute φ_B(E_A) → E_AB (same curve) # The extra points φ(P_B), φ(Q_B) were needed # for the math to work out correctly # These extra points are what killed SIKE
The Castryck-Decru Attack — 62 Minutes

Wouter Castryck and Thomas Decru (KU Leuven, July 2022) found that the auxiliary points φ(P_B), φ(Q_B) given in the public key provided enough information to reconstruct the secret isogeny using a classical mathematical result — Kani's theorem about products of elliptic curves.

# The attack insight (simplified) SIKE public key reveals: E_A (Alice's public curve) φ_A(P_B), φ_A(Q_B) ← images of Bob's torsion points Castryck-Decru showed: These extra points over-constrain the isogeny Using Kani's theorem on product curves: E × E → (E_A × E_B) factors The factorisation reveals φ_A directly # Attack cost: classical computation, ~62 minutes # No quantum computer needed # Single researcher, laptop-grade hardware # Breaks all SIKE parameter sets completely

The auxiliary points were not a bug — they were a necessary feature of how SIDH was designed to work. The security assumption was simply wrong: the auxiliary points revealed far more about the secret than anyone realised.

Why Six Years of NIST Review Missed It

SIKE was submitted in 2017, survived three rounds of NIST review, and was about to be standardised. How did the break get missed?

  • New math family — isogeny-based crypto was only ~10 years old at submission. The community analysing it was tiny compared to RSA (50 years) or lattice crypto (25 years).
  • Wrong security model — reviewers focused on finding isogenies directly. Nobody thought to use Kani's theorem — a result from a completely different area of algebraic geometry.
  • Auxiliary points not questioned deeply enough — they were seen as a clever engineering trick, not a security liability. The assumption "they're needed, so they're fine" was never rigorously challenged.
  • Cross-field insight required — the attack combined isogeny theory with results about abelian varieties and Kani's lemma — a connection nobody in the isogeny crypto community had made.
Lessons for All PQC Deployments
  • Analysis time matters enormously — LWE has 25+ years of study by thousands of researchers. ML-KEM is built on much more solid ground than SIKE's 10-year-old foundation.
  • Cryptographic agility is mandatory — systems must be able to swap algorithms quickly when one breaks. SIKE's collapse would have been catastrophic for systems locked into it.
  • Hybrid mode provides insurance — X25519 + ML-KEM means if ML-KEM falls like SIKE, X25519 still protects the session (classically). Hybrid is not paranoia — it is engineering prudence.
  • Features can be vulnerabilities — the auxiliary points were a necessary feature, not an optional extra. Security features must be scrutinised as potential attack surfaces, not just the algorithm core.
  • Math creates AND breaks security — the attack used pure classical mathematics. No quantum computer, no implementation flaw, no side-channel. The security assumption was simply wrong.

The uncomfortable truth

We cannot be certain ML-KEM will not face a SIKE-style collapse. What we can say: LWE has survived 25 years of analysis by hundreds of researchers across dozens of countries, including adversarial nation-states who have every incentive to break it and would benefit enormously from doing so silently. The security margin is orders of magnitude larger than SIKE's was. This is the strongest evidence available — not a guarantee.

Breaking the Code Layer

Implementation attacks — exploiting the gap between a mathematically secure algorithm and its real-world software or hardware realisation. This is where most practical PQC attacks will occur in the near term.

Timing Attack on ML-KEM — Concrete Example

Code Layer · Timing SCA

ML-KEM's FO transform requires a conditional check — if the re-encrypted ciphertext doesn't match, reject. A naive implementation that returns early on mismatch creates a timing oracle that leaks the secret key.

# VULNERABLE ML-KEM decapsulation def decaps_vulnerable(sk, c): m_prime = decrypt(sk, c) c_prime = encrypt(pk, m_prime) if c != c_prime: # ← timing leak here return reject_early() # returns FAST return KDF(m_prime) # returns SLOW # Attacker measures: fast=mismatch, slow=match # Crafts ciphertexts near boundary # Binary search on secret key bits # SAFE ML-KEM decapsulation def decaps_safe(sk, c): m_prime = decrypt(sk, c) c_prime = encrypt(pk, m_prime) match = ct_compare(c, c_prime) # constant time # Always compute both paths result_real = KDF(m_prime) result_dummy = random_bytes() return ct_select(match, result_real, result_dummy)

Why this is hard to get right

Constant-time programming is extremely unintuitive. Compilers optimise code — and "optimisation" often means "reintroduce the branch the programmer removed." A compiler can silently turn a constant-time compare back into an early-exit compare during optimisation.

The official ML-KEM reference implementation uses carefully verified constant-time operations. But every port to a new platform, language, or compiler must re-verify this property independently.

Real consequence: PQC-TLS timing attacks could leak session keys across the network — no physical access required.

Decryption Failure Attack on ML-KEM — Deep Dive

Code Layer · Oracle Attack

ML-KEM has a non-zero (tiny) probability of honest decryption failure — where a valid ciphertext decrypts incorrectly. If an attacker can craft ciphertexts that fail and observe the failure, they can reconstruct the secret key.

When ML-KEM Decryption Fails Honestly
# ML-KEM decapsulation computes: m' = v - s^T * u # should equal ⌊q/2⌋ * m # The noise terms: v - s^T*u = (t^T*r + e2 + ⌊q/2⌋m) - s^T*(A^T*r + e1) = (As+e)^T*r + e2 + ⌊q/2⌋m - s^T*A^T*r - s^T*e1 = ⌊q/2⌋m + e^T*r + e2 - s^T*e1 (noise terms) # Decryption fails when noise term is too large: |e^T*r + e2 - s^T*e1| > q/4 # wrong bit decoded # This happens with probability ~2^-164 for ML-KEM-768 # Vanishingly small in honest use # But attackers can craft r, e1, e2 to maximise noise
The Oracle Attack — Probing Secret Key Bits
# Attacker's strategy (without FO protection) For each target bit position i of secret key s: 1. Craft ciphertext (u*, v*) such that: Decryption failure iff s[i] > threshold 2. Submit (u*, v*) to decapsulation oracle 3. Observe: success → s[i] small failure → s[i] large 4. Binary search refines estimate of s[i] After ~1000 queries per coefficient: Full secret key s reconstructed # Key insight: attacker controls e1, e2, r # By choosing them carefully, noise term becomes # a function of exactly one coefficient of s # Failure = that coefficient exceeded threshold # = one bit of information about s per query
The FO Transform — How ML-KEM Blocks the Oracle
# FO Transform in ML-KEM decapsulation def decaps_fo(sk, c): # Step 1: decrypt (get candidate message) m_candidate = decrypt(sk.s, c) # Step 2: re-encrypt using same randomness derivation r_derived = H(m_candidate || sk.H_pk) c_check = encrypt(pk, m_candidate, r_derived) # Step 3: constant-time compare if ct_equal(c, c_check): return KDF(m_candidate || r_derived) else: return KDF(sk.z || c) # random-looking output # Why this blocks the oracle: # Attacker's crafted (u*, v*) will NOT re-encrypt # to itself — it was not produced honestly # So the FO check ALWAYS fails for crafted ciphertexts # Attacker always gets KDF(z||c) — random, no info # Oracle completely blind — no failure signal
⚠️ If the FO check is skipped, faulted, or implemented incorrectly — the full decryption oracle is exposed and the key can be recovered. This single check is the critical security boundary in ML-KEM.

Power Analysis on FALCON — The Gaussian Sampler Problem

Code Layer · Power SCA

FALCON's signing algorithm requires sampling from a discrete Gaussian distribution over NTRU lattice vectors. This Gaussian sampler is the hardest part of FALCON to implement securely — and the most powerful SCA target.

# FALCON Gaussian sampler (simplified) To sign message M: 1. Compute target vector t from M 2. Sample z from Gaussian distribution centred at lattice vector near t using secret NTRU basis (f, g, F, G) # The sampler uses floating-point arithmetic # to compute Gaussian probabilities # Different secret basis vectors → # Different floating-point operations → # Different power consumption pattern # Power trace → floating-point op sequence # → partial information about (f, g) # → after enough signatures → full key recovery

Why FALCON is harder to protect than ML-KEM

ML-KEM's operations are polynomial arithmetic — implementable in constant time using NTT with careful coding. FALCON's Gaussian sampler uses floating-point operations whose timing inherently depends on the values being computed — making true constant-time extremely difficult.

NIST's FIPS 206 specification includes guidance on implementing FALCON securely, but practical constant-time Gaussian sampling remains an active research problem. For high-security hardware, ML-DSA is often preferred over FALCON precisely for this reason.

Breaking the Protocol Layer

Even perfect math and perfect code can be defeated by bad protocol design. These attacks exploit how systems negotiate, combine, and use cryptographic primitives — not the primitives themselves.

Hybrid PQC Stripping Attack

Protocol Layer · Active Threat

When a TLS client offers a hybrid key share (X25519 + ML-KEM-768), a MitM attacker can strip the ML-KEM component — leaving only the classical X25519 exchange. The session degrades to a classically-only exchange vulnerable to HNDL.

# Hybrid stripping attack flow Client → Server: ClientHello key_shares: [X25519_key_share, MLKEM768_key_share] supported_groups: [X25519MLKEM768, X25519, P-256] MitM intercepts and modifies: key_shares: [X25519_key_share] ← ML-KEM stripped supported_groups: [X25519, P-256] ← PQC groups removed Server sees: only classical key shares offered Server responds: ServerHello with X25519 selected # Result: session uses X25519 only # Vulnerable to Shor's algorithm retroactively # Both parties believe negotiation was honest # Neither can detect the stripping without binding # Defence: Transcript binding # TLS 1.3 Finished message is MAC over full handshake # Modified ClientHello → MAC mismatch → connection drops # But: does the server enforce minimum PQC requirement? # If server accepts X25519-only → attack succeeds silently

Defence — Server-side enforcement

TLS 1.3's Finished MAC protects against transcript modification — any stripping causes a handshake failure detectable by both parties. However, the attack succeeds if the server is configured to accept classical-only fallback. The defence is server policy: reject any key exchange that does not include a PQC component. This is still not widely enforced — making hybrid stripping an active deployment risk during the transition period.

Certificate Downgrade During PQC Transition

Protocol Layer · Transition Risk

During the migration period, servers will offer both classical (RSA/ECDSA) and PQC (ML-DSA) certificates. An attacker can force selection of the classical certificate — restoring vulnerability to Shor's for certificate authentication.

# Certificate algorithm downgrade Server has two certificates: cert_classical: signed with ECDSA P-256 cert_pqc: signed with ML-DSA-65 Client Hello: "I support ML-DSA signatures" MitM modifies: "I support ECDSA only" Server selects: cert_classical (ECDSA) # Authentication now uses ECDSA # Vulnerable to Shor's algorithm # Key exchange may use ML-KEM (unaffected) # But server identity auth is compromised # MitM can forge server identity in quantum era

Why this matters even with ML-KEM

If the key exchange uses ML-KEM but the certificate is ECDSA — a quantum attacker can forge the server's certificate and impersonate the server. The client establishes a quantum-safe session with what it believes is the real server — but is actually the attacker.

Defence: Use hybrid certificates (both ECDSA and ML-DSA signatures). Clients supporting PQC must reject classical-only certificates from servers that previously offered PQC — similar to HSTS pin enforcement.

Key Management Failures

Human Layer · Operational

PQC keys are larger than classical keys — ML-KEM private keys are 2,400 bytes vs 32 bytes for Ed25519. This creates new operational challenges and new failure modes.

  • Key backup failures — larger keys mean larger backups. Teams accustomed to 32-byte Ed25519 seeds may make mistakes with 2.4KB ML-KEM private keys — truncating, corrupting, or storing insecurely.
  • HSM compatibility — many HSMs were built for RSA/ECC key sizes. PQC keys may not fit in existing HSM key slots or may require firmware updates that introduce new vulnerabilities.
  • Key rotation neglect — PQC migration often treated as "one-time task." But key rotation policies must continue — a compromised ML-KEM private key is just as dangerous as a compromised RSA key.
  • Certificate lifecycle confusion — dual-algorithm certificates (ECDSA + ML-DSA) have two separate expiry and renewal cycles. Teams may renew one and forget the other — leaving a hybrid cert with one expired component.
  • Logging private keys — debugging PQC implementations sometimes results in private key material being written to logs. ML-KEM's 2.4KB private key is large enough to be "obviously a key" — but it has happened with smaller keys.
  • Supply chain attacks — PQC libraries (libpqc, liboqs, BouncyCastle PQC) are relatively new. A compromised PQC library is far more dangerous than a compromised classical library — fewer eyes have reviewed the code.

The Dual_EC_DRBG Lesson — Backdoored Standards

Human Layer · Historical Warning

In 2006 NIST standardised Dual_EC_DRBG — a random number generator that NSA had secretly backdoored. Anyone using it generated "random" keys that NSA could predict. This remained hidden until Snowden documents confirmed it in 2013.

# Dual_EC_DRBG backdoor (simplified) Algorithm uses two curve points: P and Q "Random" output = x-coordinate of (output * Q) If NSA knows: k such that Q = k*P Then given output, NSA can compute: next_state = output * k ← predicts all future values # NSA chose P and Q — and knew k # All randomness generated was predictable by NSA # Any key, nonce, IV generated was compromised # Juniper VPN used Dual_EC → NSA backdoor active

Could this happen to PQC?

NIST's PQC process was more transparent than Dual_EC_DRBG — public submissions, public review, international participation. But the concern is legitimate and has been raised by researchers.

The LWE parameters (q, n, error distribution σ) were chosen by NIST based on NIST's analysis. If any parameters were subtly weakened — reducing security margins below stated levels — this would be extremely difficult to detect without independently verifying the security analysis.

Defence: Independent verification of all NIST parameter choices by multiple nations' cryptographic agencies (ANSSI, BSI, GCHQ). This has happened — and no evidence of backdooring has been found. But healthy scepticism is appropriate.

Cryptographic Agility — The Most Important Property

Architecture · Defence

Cryptographic agility means designing systems so that the algorithm can be replaced without redesigning the entire system. SIKE proved that no algorithm is guaranteed to survive — systems must be able to pivot quickly.

# Non-agile system (rigid — dangerous) class EncryptionService: def encrypt(self, data): return mlkem768_encrypt(data, self.key) # hardcoded # Agile system (replaceable — safe) class EncryptionService: def __init__(self, algorithm="mlkem768"): self.algo = ALGORITHM_REGISTRY[algorithm] def encrypt(self, data): return self.algo.encrypt(data, self.key) # swappable # When ML-KEM-768 is broken: # config["algorithm"] = "mlkem1024" # Deploy. Done. No redesign.

  • Negotiation in all protocols — client and server agree on algorithm, don't hardcode it

  • Regular algorithm review — schedule annual review of all cryptographic primitives in use

PQC is necessary but not sufficient

Deploying ML-KEM and ML-DSA eliminates the quantum threat to the math layer. It does nothing to fix timing vulnerabilities, fault injection targets, protocol design flaws, or weak RNG. A system using ML-KEM with a timing-vulnerable FO implementation is weaker than a system using ECDH with a constant-time implementation.

Security is only as strong as the weakest layer. PQC deployment must be accompanied by implementation review, hardware hardening for physical-access threat models, protocol enforcement, and operational security — not just an algorithm swap.

The three questions for every PQC deployment

  • Is the math right? — Use NIST-standardised algorithms (ML-KEM-768+, ML-DSA-65+). Hybrid mode during transition. Cryptographic agility built in.
  • Is the code right? — Constant-time implementations, formally verified where possible. Masking for hardware deployments. FO transform integrity verified in code review.
  • Are the people and processes right? — Hardware RNG verified. Key management updated for larger key sizes. HNDL migration started. Agility to pivot if an algorithm breaks.

Quantum vulnerability here

A VPN creates an encrypted tunnel between two endpoints — making traffic appear to come from a different location and protecting data in transit. The cryptography underneath is what matters for quantum migration.

What a VPN Actually Does — The Tunnel Model

Core Concept

Three things a VPN provides

  • Confidentiality — traffic between you and VPN server is encrypted. ISP sees only encrypted packets to the VPN server.
  • Integrity — tampered packets are detected and dropped via HMAC or AEAD tag.
  • IP masking — destination sees VPN server's IP, not yours. Geographic IP location is the VPN server's location.

A VPN does NOT provide: anonymity from the VPN provider, protection from malware on your device, or security after the traffic leaves the VPN server unencrypted.

TLS History — From SSL to TLS 1.3

TLS did not appear fully formed. It evolved through six major versions over 30 years — each one fixing the vulnerabilities that broke the previous one. Understanding the history explains why TLS 1.3 is designed the way it is.

Why each version was broken

SSL 2.0 — weak export ciphers mandated by US government + MAC applied to plaintext not ciphertext (MAC-then-encrypt flaw). SSL 3.0 — POODLE: attacker forces fallback to SSL 3.0, exploits CBC padding oracle. TLS 1.0/1.1 — BEAST: predictable CBC IV allows chosen-plaintext attack. TLS 1.2 — mostly safe with correct config, but allowed weak cipher suites (RC4, 3DES, export ciphers, RSA key exchange without forward secrecy). TLS 1.3 — eliminated all of this by removing weak algorithms entirely, not just making them optional.

What TLS 1.3 Removed — And Why

Deliberate Removals

Removed cipher suites:

  • RC4 — statistical biases, broken
  • 3DES — SWEET32 birthday attack
  • AES-CBC — padding oracle attacks
  • Export cipher suites — FREAK, LOGJAM
  • NULL cipher — no encryption at all
  • MD5 and SHA-1 based MACs — collision attacks

Removed key exchange mechanisms:

  • RSA key exchange — no forward secrecy, one key compromise decrypts all past sessions
  • Static DH — no forward secrecy
  • Anonymous DH/ECDH — no authentication, trivial MitM
  • EXPORT key exchange — artificially weakened
  • SRP — complex, rarely used
  • DSA — replaced by ECDSA/EdDSA

The key insight behind TLS 1.3 design

TLS 1.2 allowed weak algorithms and relied on correct configuration to exclude them. In practice, misconfigured servers left weak options enabled. TLS 1.3 solved this by removing weak algorithms from the protocol entirely — there is no configuration option to re-enable them. You cannot accidentally enable RC4 in TLS 1.3 because it does not exist in TLS 1.3.

HelloRetryRequest — when key share mismatches

If the client did not send a key share for the server's preferred group, the server sends a HelloRetryRequest asking the client to resend with the correct group. This adds 1 RTT — making the handshake 2 RTT total. This is why clients send a key share for the most likely group (X25519) preemptively — to avoid the retry.

TLS 1.3 Key Schedule — Full Derivation

HKDF Key Derivation

TLS 1.3 uses HKDF (HMAC-based Key Derivation Function) to derive all keys from the ECDHE shared secret. The key schedule is a carefully ordered sequence that produces independent keys for every purpose.

# TLS 1.3 Key Schedule (RFC 8446 Section 7.1) PSK = 0 (no pre-shared key) or actual PSK for resumption Early Secret = HKDF-Extract(salt=0, IKM=PSK) ↓ Handshake Secret = HKDF-Extract(salt=Early_Secret, IKM=ECDHE_shared) ↓ Master Secret = HKDF-Extract(salt=HS, IKM=0) # From Handshake Secret — used during handshake client_handshake_traffic_secret = HKDF-Expand(HS, "c hs traffic", H(CH+SH)) server_handshake_traffic_secret = HKDF-Expand(HS, "s hs traffic", H(CH+SH)) client_hs_key = HKDF-Expand(c_hs_traffic, "key", L) ← AES-256 key client_hs_iv = HKDF-Expand(c_hs_traffic, "iv", 12) ← GCM nonce base server_hs_key = HKDF-Expand(s_hs_traffic, "key", L) server_hs_iv = HKDF-Expand(s_hs_traffic, "iv", 12) # From Master Secret — used for application data client_app_traffic_secret = HKDF-Expand(MS, "c ap traffic", H(full_handshake)) server_app_traffic_secret = HKDF-Expand(MS, "s ap traffic", H(full_handshake)) client_app_key = HKDF-Expand(c_ap_traffic, "key", L) ← actual HTTPS key server_app_key = HKDF-Expand(s_ap_traffic, "key", L) client_app_iv = HKDF-Expand(c_ap_traffic, "iv", 12) server_app_iv = HKDF-Expand(s_ap_traffic, "iv", 12) # exporter_master_secret — for application-level key export # resumption_master_secret — for session resumption tickets # H = transcript hash of all handshake messages so far # L = key length (16 for AES-128, 32 for AES-256)

Why the key schedule is designed this way

Every derived key is bound to the full handshake transcript hash H. This means keys are different for every session — even if the same ECDHE keys were reused (which they are not). The hierarchical structure means compromising one derived key does not reveal others. Application keys (used for your HTTPS data) are derived after the full handshake is verified — ensuring no key is used if the handshake was tampered with.

Finished Message — Handshake Integrity Proof

Binding the Handshake
# Finished message — HMAC over entire handshake transcript finished_key = HKDF-Expand(server_hs_traffic, "finished", hash_length) server_Finished = HMAC( finished_key, transcript_hash ← SHA-384 of ALL messages from ClientHello to Certificate ) # Client verifies: recomputes transcript hash, checks HMAC # If ANY byte was modified in ANY previous message → HMAC fails # → Handshake aborted — connection dropped # Client then sends its own Finished: client_Finished = HMAC( HKDF-Expand(client_hs_traffic, "finished"), transcript_hash_including_server_Finished ) # Server verifies client Finished # Both: confirmed handshake integrity # Keys derived from FULL verified transcript # → downgrade attack impossble (any modification = failure)

Why Finished blocks downgrade attacks

The Finished HMAC covers the entire handshake transcript — including the ClientHello that listed cipher suites. If a MitM attacker removes TLS 1.3 from the supported versions list (forcing TLS 1.2), the transcript changes, the HMAC fails, and the connection drops. Neither party accepts a modified handshake. This is TLS 1.3's primary protection against protocol downgrade attacks.

Session Resumption & 0-RTT

Resumption allows subsequent connections to skip the full handshake. 0-RTT goes further — sending application data with the very first packet, before any server response. Both have important security tradeoffs.

Session Resumption — 1-RTT with PSK

Performance Optimisation
# First connection: full handshake Client ↔ Server: full TLS 1.3 handshake (1 RTT) Server → Client: NewSessionTicket (PSK) Client stores: (ticket, PSK, server_identity) # Second connection: PSK resumption (still 1 RTT) Client → Server: ClientHello + { pre_shared_key: [ticket_identity, binder] psk_key_exchange_modes: [psk_dhe_ke] ← requires fresh ECDHE key_share: <fresh ECDHE key> } Server → Client: ServerHello + { pre_shared_key: selected_identity key_share: <server ECDHE key> } # NO Certificate, NO CertificateVerify # Server identity assumed from PSK # Still 1 RTT — same latency as full handshake # But: less compute (no cert chain verify)

Why PSK still uses ECDHE

psk_dhe_ke mode (not psk_ke) requires a fresh ECDHE key exchange alongside the PSK. This maintains forward secrecy — compromising the PSK does not reveal past sessions because each session also used an independent ephemeral ECDHE secret.

psk_ke (PSK without ECDHE) is also defined but loses forward secrecy — avoid it except for constrained IoT where the compute cost of ECDHE is prohibitive.

Benefit of resumption: Skips certificate validation (expensive chain verification + OCSP check) → 40-60% CPU reduction on the server for repeat visitors.

0-RTT Early Data — Maximum Performance, Maximum Risk

Performance vs Security

0-RTT allows the client to send application data with the very first packet — before receiving any server response. The web page request arrives at the server simultaneously with the ClientHello. Eliminates handshake latency entirely for repeat visitors.

How 0-RTT Works
# 0-RTT flow # Previous connection issued: NewSessionTicket with max_early_data # New connection — client sends everything at once Client → Server: [ ClientHello + key_share + pre_shared_key + early_data extension early_data: AES-256-GCM(early_traffic_key, HTTP_GET_request) ] # Early traffic key derived from PSK alone (no ECDHE yet) early_secret = HKDF-Extract(0, PSK) client_early_key = HKDF-Expand(early_secret, "c e traffic", H(CH)) # Server receives ClientHello + HTTP request simultaneously # Server decrypts early data using same key derivation # Server can start processing the HTTP request immediately Server → Client: [ ServerHello + key_share EncryptedExtensions (early_data: accepted) Certificate + CertificateVerify + Finished HTTP response (already being prepared from early data) ] # Result: HTTP response arrives before handshake fully complete # Perceived latency: 0 RTT for repeat visitors

The trust infrastructure that makes TLS authentication meaningful. Without PKI, TLS encryption is useless — you could be encrypted with an attacker. PKI solves identity. Certificate Transparency solves rogue CA certificates.

QUIC & HTTP/3 — TLS Inside a New Transport

HTTP/3 runs over QUIC — a new transport protocol that bakes TLS 1.3 directly into the connection establishment. No separate TCP handshake, no separate TLS handshake — they happen simultaneously. The fastest possible start.

TCP+TLS vs QUIC — The Latency Comparison

Why QUIC Exists
# TCP + TLS 1.3 (HTTP/2) RTT 0: Client SYN → RTT 0: ← Server SYN-ACK (TCP handshake) RTT 0: Client ACK → RTT 1: Client Hello → RTT 1: ← Server Hello + Cert + Finished (TLS) RTT 1: Client Finished → RTT 1: HTTP GET → ← same packet as Finished RTT 2: ← HTTP Response Total: 2 RTTs before first byte of response (or 1 RTT if you count GET with Finished) # QUIC + TLS 1.3 (HTTP/3) RTT 0: QUIC Initial + ClientHello + HTTP GET → RTT 1: ← QUIC Initial + ServerHello + Cert + Finished + HTTP Response Total: 1 RTT before first byte of response (or 0 RTT on resumption)

Problems QUIC solves

  • Head-of-line blocking — TCP: lost packet blocks ALL streams. QUIC: lost packet only blocks that one stream. HTTP/3 with QUIC eliminates HoL blocking entirely.
  • Connection migration — QUIC uses Connection IDs not IP+port. Switch from WiFi to cellular — connection survives without reconnection.
  • Faster handshake — crypto built in, no separate TCP + TLS sequence.
  • Better mobile performance — mobile networks have high packet loss. QUIC's independent streams + connection migration help significantly.

Heartbleed — CVE-2014-0160

Not a cryptographic break — a buffer over-read in OpenSSL's TLS heartbeat extension. Exposed private keys, passwords, and session tokens from ~500,000 HTTPS servers simultaneously. Discovered April 2014.

What Happened — The Exact Vulnerability

Memory Safety · Code Layer
# TLS Heartbeat — RFC 6520 # Purpose: keep-alive ping — "are you still there?" Client → Server: HeartbeatRequest { payload_length: 500 ← attacker sets this payload: "hello" ← actual data: 5 bytes } Server → Client: HeartbeatResponse { payload: server_memory[0..500] ← BUG: copies 500 bytes! } # The vulnerable OpenSSL code (simplified) int tls1_process_heartbeat(SSL *s) { unsigned char *p = &s->s3->rrec.data[0]; unsigned int payload = *(unsigned short*)(p + 1); ← trusts client value unsigned char *pl = p + 3; ← no bounds check! memcpy(bp, pl, payload); ← reads payload bytes # payload=500, actual data=5 → reads 495 bytes PAST the buffer # those 495 bytes = server RAM contents: keys, passwords, tokens } # Fix (one line): check payload ≤ actual received length # if (payload + 3 > s->s3->rrec.length) return 0;

What attackers extracted

  • Server private keys (SSL/TLS private key — could impersonate server forever)
  • Session tokens (hijack active logged-in sessions)
  • Username/password pairs (from recent logins in memory)
  • Secret keys for symmetric encryption (decrypt past/future sessions)
  • Any data in server RAM at time of attack (memory contents are random)

Scale and impact

  • ~17% of all HTTPS servers affected (Netcraft: ~500,000 servers)
  • Yahoo, LastPass, Imgur, OKCupid confirmed compromised
  • 40+ million OpenSSL installations needed patching within 24 hours
  • Certificate revocation crisis — ~500,000 certs needed revoking simultaneously, breaking OCSP infrastructure
  • Canada Revenue Agency: 900 SINs stolen — government shutdown of e-filing
Prevention — One Fix, One Process Change
  • Technical fix: Bounds check — verify payload_length ≤ actual received data length before memcpy. One line of code. The fix took 2 hours to write and test.
  • Process fix: Memory-safe languages (Rust, Go) make this class of bug impossible. OpenSSL written entirely in C with manual memory management.
  • Detection: Fuzzing would have found this — libFuzzer on the heartbeat path would have triggered the over-read within seconds.
  • Funding: OpenSSL had ~$2,000/year in donations when Heartbleed hit. The Core Infrastructure Initiative was founded immediately after — $3.6M/year to critical open-source security libraries.

POODLE & BEAST — CBC Padding Oracle Attacks

Two related attacks on CBC mode in SSL/TLS. BEAST (2011) — chosen plaintext on TLS 1.0. POODLE (2014) — padding oracle on SSL 3.0 fallback. Both exploited the fundamental flaw in how CBC mode was used in SSL/TLS.

CBC Padding Oracle — The Root Cause

Cryptographic Flaw · Math + Protocol
# CBC decryption P[i] = Decrypt(C[i]) XOR C[i-1] # PKCS#7 padding — last block must be padded # to block boundary with padding bytes = padding length ... | data | 03 03 03 | ← 3 bytes of padding ... | data | 05 05 05 05 05 | ← 5 bytes # Server decrypts, checks padding validity # If padding invalid → "padding error" response # If padding valid → "MAC error" response (different!) # This difference is the oracle # Attacker can distinguish valid from invalid padding # via timing, error message, or connection close behavior

The Attack Mechanism

By modifying the previous ciphertext block (C[i-1]) and observing whether decryption produces valid padding, an attacker can learn the plaintext one byte at a time — 256 attempts per byte maximum.

For a 16-byte cookie: 16 × 128 = ~2,048 requests on average. Automated tools recover session cookies in minutes. POODLE forced SSLv3 fallback (every browser supported SSLv3 for backward compatibility) to exploit this.

POODLE (SSL)2014SSL 3.0 CBCDowngrade to SSL 3.0, exploit padding oracle — cookie recovery in <256 requests/byteDisable SSL 3.0 entirely — TLS_FALLBACK_SCSV

Prevention
  • Use AEAD modes (AES-GCM, ChaCha20-Poly1305) — authenticated encryption authenticates before decrypting, making padding oracles impossible. GCM has no padding.
  • Encrypt-then-MAC instead of MAC-then-Encrypt (CBC with HMAC) — attacker cannot forge valid ciphertext to probe padding.
  • Disable all SSLv3 — TLS_FALLBACK_SCSV prevents deliberate downgrade to SSLv3.
  • TLS 1.3 — removed CBC entirely. All cipher suites are AEAD. POODLE is architecturally impossible in TLS 1.3.

ROBOT — Return Of Bleichenbacher's Oracle Threat

A 1998 attack on RSA PKCS#1 v1.5 encryption — resurfaced in 2017 and found on major production servers including Facebook, PayPal, Cisco, and dozens of others. The same 19-year-old vulnerability, still deployed.

Bleichenbacher's Attack — RSA Padding Oracle

Math + Protocol · 1998/2017
# RSA PKCS#1 v1.5 encryption format EM = 0x00 || 0x02 || PS || 0x00 || M # PS = random non-zero padding (≥8 bytes) # M = actual message (pre-master secret) # Server decrypts RSA ciphertext, checks: # Does EM start with 0x00 0x02? # If YES: "conformant" (implicit signal) # If NO: "non-conformant" (different handling) # Bleichenbacher 1998: use this as oracle # Multiply ciphertext by s: C' = C × sᵉ mod n # Server decrypts C': is it conformant? # Binary search on M using conformance signal # After ~1 million queries: recover M completely # M = TLS pre-master secret = decrypt all past sessions

Why it resurfaced in 2017

Many implementations fixed the 1998 attack but introduced subtle timing differences — fast path for conformant, slow path for non-conformant. The oracle was still there, just quieter. Böck, Somorovsky, and Young found it on: F5 BIG-IP, Cisco ACE, Citrix NetScaler, Radware, and others. Facebook's main domain was vulnerable. PayPal was vulnerable.

19 years. Same vulnerability. Still deployed on major production infrastructure.

Prevention
  • Disable RSA key exchange entirely — use ECDHE only. RSA key exchange has no forward secrecy AND enables Bleichenbacher. No legitimate reason to use it in 2025.
  • RSA-OAEP for encryption — if RSA encryption is needed (not key exchange), use OAEP padding not PKCS#1 v1.5.
  • TLS 1.3 — removed RSA key exchange entirely. ROBOT is architecturally impossible in TLS 1.3.
  • Constant-time RSA decryption — if PKCS#1 v1.5 must be used (legacy), the conformance check must take identical time regardless of result.

SHAttered — SHA-1 Collision Attack

February 2017: Google and CWI Amsterdam produced the first practical SHA-1 collision — two different PDF files with identical SHA-1 hashes. The attack cost ~110 GPU-years and ~$110,000. Math breaking math, on schedule.

The Attack — Two PDFs, One Hash

Cryptanalysis · Math Layer
# SHAttered result SHA1(PDF_1) = 38762cf7f55934b34d179ae6a4c80cadccbb7f0a SHA1(PDF_2) = 38762cf7f55934b34d179ae6a4c80cadccbb7f0a # PDF_1 and PDF_2 are different files # Same SHA-1 hash — collision found # Attack complexity: 9.2 × 10^18 SHA-1 compressions 110 GPU-years total computation Cost estimate: ~$110,000 (2017 GPU prices) or $45,000 on AWS spot instances # Theoretical birthday bound: 2^80 operations # Actual attack: 2^63.1 — differential cryptanalysis # 100,000× cheaper than birthday attack # Structural weakness exploited, not brute force

Why it mattered beyond PDFs

  • Git uses SHA-1 — a collision could produce two different commits with the same hash. Repository integrity guarantee broken. Git is migrating to SHA-256.
  • TLS certificates — SHA-1 cert signatures had already been deprecated (browsers stopped accepting in 2017 — timing was not coincidental).
  • Code signing — software signed with SHA-1 could be replaced with malicious binary sharing the same hash.
  • Digital signatures — any signature scheme using SHA-1 is breakable: attacker creates two documents, gets signature on benign one, transfers to malicious one.
Prevention
  • Never use SHA-1 for any security purpose — SHA-256 minimum. SHA-384 for high-security. The collision resistance of SHA-256 is 2^128 — no practical attack exists or is foreseeable.
  • The timeline was predictable: SHA-1 theoretical weakness known since 2005 (Wang et al.). Google's attack was not surprising — it was expected. Anyone still using SHA-1 in 2017 had 12 years of warning.
  • Deprecation must be proactive: Browsers deprecated SHA-1 certificates before SHAttered — the right call. Waiting for practical attack before deprecating is too late.

  • Weak internal security controls despite being a trusted CA
  • No automated revocation system — manual revocation took too long
  • DigiNotar knew about breach for days before disclosing — ethically and legally wrong

Juniper Networks — NSA Backdoor (2015)

Backdoor · Supply Chain

Juniper Networks discovered unauthorized code in their ScreenOS firewall firmware. Two separate backdoors: one allowing SSH login with a hardcoded password, another decrypting VPN traffic. The VPN backdoor exploited the Dual_EC_DRBG weakness — an NSA backdoor, re-exploited by unknown actors.

# What was found in Juniper ScreenOS Backdoor 1: SSH password "<<< %s(un='%s') = %u" → admin access to any Juniper device → trivially exploitable once discovered Backdoor 2: Dual_EC_DRBG parameters changed Original NSA backdoor: NSA knows secret k for Q=k*P Juniper change: DIFFERENT k inserted for different group → unknown actor (not NSA) could decrypt all Juniper VPN traffic → affected: ~all enterprise VPNs using Juniper ScreenOS # Duration: backdoor present in firmware from 2012–2015 # Discoverer: Juniper internal code audit # Likely attacker: still unknown (China? Another nation-state?)

The deeper lesson

The Juniper backdoor was possible because Dual_EC_DRBG (a known-suspicious RNG) was already present in the firmware. Someone who knew or found the NSA's backdoor parameters modified the firmware to replace them with their own parameters — stealing the backdoor. This demonstrates: any intentional cryptographic weakness can be exploited by parties other than those who created it. There is no "secure backdoor."

Equifax — Unpatched TLS + Certificate Failure (2017)

Operational Failure · Human Layer

147 million people's financial records stolen. Not a cryptography break — a combination of unpatched Apache Struts, and critically: Equifax had not renewed a TLS inspection certificate for 19 months, meaning malicious outbound traffic was not monitored.

# Equifax breach timeline March 2017: Apache Struts RCE vulnerability published (CVE-2017-5638) March 2017: Equifax IT notified — patch not applied May 13, 2017: Attacker exploits unpatched Struts May-July 2017: Data exfiltrated over 76 days July 29, 2017: Equifax discovers breach How? TLS inspection certificate expired 19 months ago → encrypted outbound traffic invisible to security tools → exfiltration undetected for 76 days September 2017: Public disclosure 2019: FTC: $575 million settlement

Lessons

  • Patch management: Critical vulnerability + 2 months + no patch = 147M records stolen. Patch immediately.
  • Certificate lifecycle: Expired TLS inspection cert meant the security team was blind for 19 months. Certificate expiry monitoring is not optional.
  • Encryption cuts both ways: TLS protects legitimate traffic AND attacker exfiltration. TLS inspection (with proper certificate management) is essential in high-security environments.

The Meta-Pattern

Core Insight

Attacks almost never break the math

Of all the case studies in this session and Session 3, only SHAttered and SIKE broke the underlying mathematical assumption. Every other attack — Heartbleed, POODLE, ROBOT, LOGJAM, FREAK, DigiNotar, Juniper, Equifax — was a failure of implementation, protocol design, operations, or policy.

This means: switching to PQC algorithms solves the quantum math threat. It does not solve any of these other eleven patterns. All eleven will still exist and still be exploited after PQC migration.

The cost of delayed deprecation

SHA-1 weakness known 2005 → SHAttered 2017 → 12 years of exposure. SSL 3.0 flawed since inception → POODLE 2014 → nearly 20 years deployed. Export ciphers mandated 1990s → FREAK/LOGJAM 2015 → 15+ years after they should have been removed.

The pattern: industry knows an algorithm is weak, delays migration for "compatibility," pays the price when the attack becomes practical. The correct model: deprecate proactively, migrate before attacks are practical. TLS 1.3 embodies this — removed weak algorithms before breaks occurred, not after.

HSM — Hardware Security Module

The Gold Standard

An HSM is a physical device that generates, stores, and uses cryptographic keys — the private key never leaves the HSM. Even with full server compromise, the private key cannot be extracted.

HSM options by tier

  • Cloud KMS (AWS KMS, Azure Key Vault, GCP KMS) — HSM-backed, API-driven, auto-scaling. Best for most applications. ~$1/key/month.
  • Dedicated cloud HSM (AWS CloudHSM, Azure Dedicated HSM) — dedicated hardware, you control the HSM partition. ~$1.50/hour. Required for FIPS 140-3 Level 3.
  • On-premises HSM (Thales, Utimaco, nCipher) — physical device in your datacenter. $20K–$100K+. Required when data cannot leave premises.
  • Software KMS (HashiCorp Vault) — keys in memory/encrypted storage. No physical tamper protection. Good for secrets management, not long-term CA keys.

Why mTLS matters

Without mTLS, any compromised service inside your network can call any other service. With mTLS, even an internal attacker must have a valid certificate for a specific service identity. A compromised "logging service" cannot call the "payment service" — it has the wrong certificate.

Service meshes (Istio, Linkerd) implement mTLS transparently — inject sidecar proxies, handle cert rotation automatically, enforce policy. Applications see plaintext; the mesh handles all crypto. This is the cleanest way to deploy at scale.

Master Principle

What security actually requires

Cryptography gives you the tools. Key management determines whether you use them correctly. Operations determines whether they stay correct under pressure. People determine whether the whole system holds under social engineering, insider threats, and the chaos of real production.

A system using AES-128-GCM with proper key management, constant-time implementation, and disciplined operations is more secure than a system using AES-256-GCM with keys in config files, no rotation, and no monitoring. The algorithm is the smallest part of real-world security.
